# Internationalization (i18n) Checklist

## Text and Strings

- [ ] All user-facing strings are externalized into resource files (JSON, XLIFF, .properties) -- no hardcoded text in code or templates
- [ ] String concatenation is not used to build sentences -- use ICU MessageFormat or equivalent for parameterized messages
- [ ] Pluralization rules use CLDR plural categories (zero, one, two, few, many, other), not simple if/else on count
- [ ] Gender-dependent text uses gender-aware message formats, not string interpolation of gendered words
- [ ] UI layouts accommodate text expansion (German ~30% longer, Finnish ~60% longer than English)
- [ ] Strings are never split across multiple keys or UI elements -- translators need full sentences for context

## Encoding

- [ ] All files, databases, and APIs use UTF-8 encoding consistently
- [ ] String length calculations use grapheme clusters, not bytes or code points (emoji, accented characters)
- [ ] String comparison and sorting use locale-aware collation (Intl.Collator, ICU collation)
- [ ] User input is normalized (NFC) before storage and comparison

## Date, Time, Number, and Currency

- [ ] Dates and times are formatted using locale-aware formatters (Intl.DateTimeFormat, DateTimeFormatter)
- [ ] All timestamps are stored in UTC and converted to the user's timezone for display
- [ ] Number formatting respects locale conventions (decimal separator, thousands grouping)
- [ ] Currency display uses ISO 4217 codes and locale-appropriate symbol placement
- [ ] Week start day and calendar system vary by locale -- use locale-aware calendar APIs

## Layout and Visual Design

- [ ] RTL (right-to-left) layouts are supported using CSS logical properties (margin-inline-start, not margin-left)
- [ ] `dir="auto"` or explicit `dir` attributes are set on elements containing user-generated content
- [ ] Font stacks include fallback fonts that support the target script (CJK, Arabic, Devanagari)
- [ ] No text is embedded in images -- all visible text is rendered as real text for translation and accessibility
- [ ] Icons and images are reviewed for cultural appropriateness and directional assumptions (arrows, hand gestures)

## Testing

- [ ] Pseudo-localization is enabled in development to catch hardcoded strings and layout overflow
- [ ] The app can switch locales at runtime without restart for manual testing
- [ ] Automated lint rules flag hardcoded strings, concatenation patterns, and non-externalized text
- [ ] At least two locales are tested in CI: one LTR (e.g., English) and one RTL (e.g., Arabic or Hebrew)
- [ ] Date, number, and currency formatting is verified with locale-specific test cases
