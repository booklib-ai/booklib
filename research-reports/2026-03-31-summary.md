## What's Worth Building

### Teach the search engine what it's looking at

**The problem it solves:** Right now, when you search BookLib for something like "null safety best practices," the search engine treats every chunk of every skill equally. A paragraph about null safety in the Kotlin skill competes with a paragraph about error handling in the Rust skill that happens to use similar words. The search doesn't *know* which skill or topic a chunk belongs to — it only sees the words.

**How it works:** Think of it like labeling boxes before putting them in a warehouse. Instead of tossing all your stuff into unlabeled boxes and hoping you find the right one later, you write "Kitchen — Utensils — Spatulas" on each box before storing it. When you search for spatulas, you go straight to the right section. A recent paper called SRAG showed that simply adding structured labels to text before the search engine processes it improved answer quality by 30% — and that's not a small number, it's statistically bulletproof.

**What it means for BookLib:** This is where BookLib has a genuine unfair advantage. BookLib's skills already have rich structure — each chunk knows which skill it belongs to, what type of content it is (core principles, anti-patterns, framework), and what language or domain it covers. That metadata is already there in the XML tags. Flat markdown skills on platforms like skills.sh simply don't have this. By feeding that structure into the search engine, BookLib's retrieval becomes meaningfully better than competitors' — using data we already own.

**What we'd actually build:** A roughly 20-line change in the indexer. Before generating an embedding for each chunk, prepend a tag like `[skill:effective-kotlin] [type:anti_patterns] [tags:null-safety,kotlin]`. That's it. One day of work, one re-index, and search quality jumps.

**Verdict:** ✅ This is the single highest-value change we can make right now. No new dependencies, no model swaps, no risk.

### Get credit for what we already built

**The problem it solves:** BookLib's architecture — skills as persistent knowledge, specialized reviewer agents, a knowledge graph for deep context — is genuinely novel. But in a crowded market, "novel" doesn't matter if nobody knows. There's no academic or industry language to describe what BookLib does, which makes it hard to explain to enterprises, researchers, or potential contributors why it's different.

**How it works:** A February 2026 paper called "Codified Context" formalized exactly the pattern BookLib uses. The researchers built a system with three layers: "hot memory" (always-on conventions and rules), "domain expert agents" (specialized reviewers), and "cold memory" (a knowledge base of specs and docs). They tested it across 283 real development sessions and showed it prevents AI assistants from losing coherence and repeating mistakes. That's BookLib. Skills are the hot memory. The 8 reviewer agents are the domain experts. The knowledge graph is the cold memory.

**What it means for BookLib:** This is free positioning. By citing this paper and using its terminology, BookLib can describe itself in language that researchers and enterprise teams immediately understand. Instead of "we have skills and agents," it becomes "BookLib is an open-source implementation of the codified context infrastructure pattern." That sounds — and is — much more substantial.

**What we'd actually build:** Nothing in code. Update the README and the upcoming RESEARCH.md to reference the paper and map BookLib's architecture to the paper's taxonomy. About two hours of writing.

**The catch:** It's a single-author preprint, not peer-reviewed yet. Use "aligns with" rather than "validated by" when citing it.

**Verdict:** ✅ A two-hour investment that upgrades how the entire project is perceived.

### Make the search engine read the whole chapter first

**The problem it solves:** When BookLib indexes a skill, it chops it into chunks and processes each one separately. The problem is that a chunk in the middle of a document loses the context from everything around it. A paragraph that says "this approach" doesn't know what "this" refers to, because the surrounding paragraphs were processed independently.

**How it works:** Imagine you're summarizing a book. You could read each paragraph in isolation and write a summary of it. Or you could read the entire chapter first, then go back and summarize each paragraph — your summaries would be much better because you understand the full context. That's exactly what **late chunking** does. Instead of processing each chunk alone, the AI reads the entire skill file first, building a rich understanding of every word in context, and *then* splits it into chunks. Research shows this improves retrieval by about 24%.

**What it means for BookLib:** No CLI skill tool does this yet. It would be a genuine first-mover advantage. And the model needed to do it (Jina's small embedding model) already runs in the same JavaScript framework BookLib uses.

**What we'd actually build:** Swap the embedding model and add custom pooling logic in the indexer. It requires re-indexing the entire library.

**The catch:** This is more complex than it sounds. The current JavaScript library doesn't support late chunking out of the box — we'd need to write custom code to extract token-level embeddings and pool them manually. It also means switching embedding models, which breaks all existing indexes. This is a 3-5 day project, best tackled after the hybrid search pipeline is stable.

**Verdict:** ⚠️ Real and valuable, but save it for later. Don't rush it — do the metadata prefixes first, then add this on top.

### Make BookLib discoverable by every AI tool on the planet

**The problem it solves:** The MCP ecosystem (the protocol that lets AI tools talk to external services) is growing fast — over 1,000 servers, adopted by Anthropic, OpenAI, Microsoft, Google, and Amazon. But there's no standard way for tools to discover what an MCP server can do without connecting to it first.

**How it works:** The MCP project is introducing "Server Cards" — think of them like a business card for AI servers. You publish a small JSON file at a standard web address, and any registry, crawler, or AI agent can instantly see what your server offers without needing to connect. It's like putting your restaurant on Google Maps instead of hoping people walk by.

**What it means for BookLib:** BookLib already plans a `.well-known/skills/` endpoint for Spec 1. Adding an MCP Server Card alongside it means BookLib's MCP server shows up in the entire MCP discovery ecosystem — not just skill-specific tools. It's a half-day of work that dramatically increases visibility.

**What we'd actually build:** Generate one extra JSON file in the CI pipeline, following the draft specification. Host it on GitHub Pages alongside the existing skill index.

**The catch:** The Server Card spec isn't finalized yet — it's an active proposal. The format might change, but the underlying pattern (`.well-known` metadata) is stable. Even if details shift, the work is reusable.

**Verdict:** ✅ Quick win. Half a day of work for ecosystem-wide visibility.

### Organize search results by how knowledge connects

**The problem it solves:** When BookLib's knowledge graph is active, search results come back as a flat ranked list. But knowledge isn't flat — concepts connect to each other in meaningful ways. If you search for "dependency injection" and the graph knows it relates to "inversion of control" and "factory pattern," those connected results should arrive in a logical order, not randomly scattered.

**How it works:** A paper published at NAACL 2025 (a top NLP research conference) showed that after you retrieve relevant chunks, you can use a knowledge graph to organize them by how closely connected they are — closest relationships first, then more distant ones. It's like how a good teacher explains a concept starting with the foundations and building up, rather than jumping between unrelated points.

**What it means for BookLib:** BookLib already has a knowledge graph. This would make the `booklib context` command output more coherent — the AI receiving the context would get information in a logical sequence rather than a relevance-score jumble.

**What we'd actually build:** After graph expansion, sort results by graph distance, then by edge type priority. About a day of work in the graph injector module.

**The catch:** This only works well when the knowledge graph has enough nodes and connections. BookLib's graph is still young and manually curated via `booklib capture`. With a sparse graph, there simply aren't enough connections to organize. This becomes valuable as the graph grows — not right now.

**Verdict:** ⚠️ Good idea, wrong time. Revisit once Spec 3 has built up meaningful graph density.
