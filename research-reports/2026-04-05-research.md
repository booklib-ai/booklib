## Finding 1: SkillRouter — Retrieve-and-Rerank Skill Selection at 80K Scale

**Maps to:** Spec 2 (retrieval quality), `lib/engine/searcher.js`, `lib/engine/query-expander.js`
**Impact:** High | **Effort:** Medium

**Paper:** [SkillRouter: Skill Routing for LLM Agents at Scale](https://arxiv.org/abs/2603.22455) (March 23, 2026; v4 April 1, 2026)
**Source type:** Preprint — 11 authors (YanZhao Zheng, ZhenTao Zhang, Chao Ma, et al.)
**Credibility:** 4/5 — large-scale evaluation on 80K skills, tested with 4 production coding agents (Kimi-K2.5, glm-5, Claude Sonnet 4.6, Claude Opus 4.6), builds on SkillsBench benchmark, v4 revision indicates active refinement. Not yet peer-reviewed.

**Summary:** SkillRouter addresses the problem of selecting the right skill from a large registry (80K candidates) given a user task. It uses a two-stage retrieve-and-rerank pipeline: Stage 1 uses a fine-tuned 0.6B embedding model (Qwen3-Emb-0.6B) to reduce 80K skills to top-20 via cosine similarity; Stage 2 applies a 0.6B cross-encoder reranker (Qwen3-Reranker-0.6B) for final ranking. Total: 1.2B parameters. Key results: **74.0% Hit@1** routing accuracy, **13× fewer parameters** than the strongest baseline, **5.8× faster**. The paper establishes a critical insight called "hidden-body asymmetry": the routing component needs access to full skill text (name + description + body) for accurate selection, but the agent that uses the skill only sees name + description due to context window constraints. Removing body text causes a **31–44 percentage point drop** in routing accuracy.

**Training methodology highlights directly applicable to BookLib:**
1. **Listwise cross-entropy loss** outperforms pointwise binary cross-entropy by +30.7pp Hit@1 — critical for comparative scoring in homogeneous skill pools
2. **False negative filtering** via 3-layer pipeline (exact name match → body Jaccard >0.6 → embedding cosine >0.92) contributes +4.0pp
3. **Hard negative mining from 4 sources:** semantic negatives (top-50 similar), lexical negatives (BM25), taxonomy negatives (same category), random negatives — 10 negatives per query
4. **37,979 skill-routing query pairs** used for fine-tuning

**Why it matters for BookLib:** This is the most directly relevant paper to BookLib's retrieval architecture to date. Specific implications:

1. **BookLib IS a skill router.** `booklib context` retrieves relevant skills and injects them into agent context. SkillRouter's architecture (embed → retrieve top-k → rerank → inject name+description) maps directly onto Spec 2's planned hybrid pipeline.
2. **Hidden-body asymmetry validates BookLib's full-text indexing.** BookLib indexes full skill body text for retrieval but can inject progressively (--compact mode). SkillRouter proves this asymmetry is essential, not just convenient.
3. **Hard negative mining strategy is adoptable.** BookLib's Spec 2 eval harness should use the 4-source hard negative strategy (semantic, lexical, taxonomy, random) for training/evaluating the retrieval pipeline. The taxonomy negatives connect to the taxonomy frontmatter field (already in IDEAS.md backlog).
4. **Listwise loss finding informs reranker choice.** When BookLib adds cross-encoder reranking, the loss function matters enormously. If fine-tuning a reranker, use listwise loss.
5. **The 80K skill scale anticipates BookLib's growth.** Current: ~280 discoverable skills. As the ecosystem grows, SkillRouter's techniques become essential.

**Actionable implementation:**
- In `lib/engine/searcher.js`, ensure the retrieval pipeline accesses full skill body text (not just frontmatter) for scoring — validated by the 31-44pp drop without body
- For Spec 2 eval harness in `benchmark/`, adopt the 4-source hard negative mining strategy
- When building the cross-encoder reranker, prioritize listwise training over pointwise if fine-tuning
- Reference SkillRouter's false negative filtering (Jaccard + cosine dedup) as a pattern for `booklib audit --dedup` to detect near-duplicate skills in registries

## Finding 2: BGE-reranker-v2-m3 ONNX — Production-Ready Cross-Encoder for Transformers.js

**Maps to:** Spec 2 (cross-encoder reranking), `lib/engine/reranker.js` (new)
**Impact:** High | **Effort:** Low-Medium

**Source:** [mogolloni/bge-reranker-v2-m3-onnx](https://huggingface.co/mogolloni/bge-reranker-v2-m3-onnx) — ONNX export of [BAAI/bge-reranker-v2-m3](https://huggingface.co/BAAI/bge-reranker-v2-m3)
**Source type:** Official BAAI model (peer-reviewed research lab, MTEB-benchmarked) + community ONNX conversion
**Credibility:** 4/5 — BAAI is a top-tier research institution, bge-reranker-v2-m3 is widely adopted and benchmarked. ONNX export verified with working Node.js example code. Apache 2.0 license.

**Summary:** BAAI's bge-reranker-v2-m3 is a multilingual cross-encoder reranker now available as a **confirmed working ONNX model compatible with @huggingface/transformers (transformers.js) in Node.js**. The model takes query-passage pairs as input and outputs raw relevance logits. Key specs: fp32 format, ~2.2GB model size, CPU-compatible, Apache 2.0 license. A working Node.js code example exists using `AutoModelForSequenceClassification` and `AutoTokenizer` from `@huggingface/transformers` — the exact library BookLib already uses.

**Why it matters for BookLib:** This **unblocks Spec 2's cross-encoder reranking**. Previous reports identified the SEE early-exit cross-encoder (04-01) as backlogged because there was "no ONNX path" for cross-encoders in transformers.js. BGE-reranker-v2-m3-ONNX provides that path:

1. **Drop-in compatible with BookLib's stack.** Uses `@huggingface/transformers` which BookLib already depends on. No new runtime dependencies.
2. **Proven reranking quality.** BGE-reranker-v2-m3 outperforms most cross-encoders in its class on MTEB retrieval benchmarks.
3. **Multilingual support** covers BookLib's international user base.
4. **The 2.2GB model size is the main concern.** For a CLI tool, downloading 2.2GB on first use is significant. Mitigation: lazy download on first `booklib search --rerank` invocation, with a flag to opt-in.

**Also notable:** Qwen3-Reranker-0.6B-seq-cls ([tomaarsen/Qwen3-Reranker-0.6B-seq-cls](https://huggingface.co/tomaarsen/Qwen3-Reranker-0.6B-seq-cls)) is a newer, higher-quality reranker (outperforms BGE-reranker-v2-m3 on MTEB-R and MTEB-Code) converted to sequence classification format. However, it does NOT yet have an ONNX export or confirmed transformers.js compatibility. At 0.6B params it would also be significantly larger. Monitor for ONNX conversion — if it appears, it would be the preferred reranker for code-specific tasks (73.42 MTEB-Code vs BGE's lower score).

**Actionable implementation:**
1. Create `lib/engine/reranker.js` with BGE-reranker-v2-m3-ONNX as the cross-encoder backend
2. Pipeline: `searcher.js` retrieves top-20 via BM25+vector → `reranker.js` re-scores and returns top-5
3. Lazy model download: first invocation of `booklib search --rerank` triggers download with progress bar
4. Add `--no-rerank` flag to skip reranking for latency-sensitive use cases
5. Benchmark latency: cross-encoder scoring of 20 query-passage pairs should be <500ms on CPU for CLI acceptability

**Files:** `lib/engine/reranker.js` (new), `lib/engine/searcher.js`, `bin/cli.js`, `package.json`

## Finding 3: SkillReducer — 48% Skill Compression with Quality Improvement

**Maps to:** Core (`booklib audit`), skill authoring guidelines, Spec 1 (skill format)
**Impact:** High | **Effort:** Low

**Paper:** [SkillReducer: Optimizing LLM Agent Skills for Token Efficiency](https://arxiv.org/abs/2603.29919) (March 31, 2026)
**Source type:** Preprint — 6 authors (Yudong Gao, Zongjie Li, Yuanyuan Yuan, Zimo Ji, Pingchuan Ma, Shuai Wang)
**Credibility:** 4/5 — large-scale analysis of 55,315 public skills, evaluated on SkillsBench (600 skills), 0.965 transfer rate across 5 models from 4 families. Not yet peer-reviewed but rigorous methodology with adversarial testing.

**Summary:** SkillReducer presents the first systematic analysis of token efficiency in agent skills. A study of 55,315 public skills found systemic problems: **26.4% lack routing descriptions entirely**, **over 60% of body content is non-actionable** (verbose explanations, examples that repeat rules, filler text), and reference files can inject **tens of thousands of tokens** per invocation. The paper introduces a two-stage optimization framework: (1) adversarial delta debugging for description compression — iteratively removes tokens and tests whether routing accuracy degrades; (2) taxonomy-driven classification + progressive disclosure for body restructuring — categorizes body content as "core rules," "supporting examples," "reference material" and applies compression per category. Results: **48% description compression**, **39% body compression**, and a **+2.8% functional quality improvement** — confirming a "less-is-more" effect where removing non-essential content reduces attention dilution.

**Why it matters for BookLib:** This paper provides quantitative methodology for three BookLib features:

1. **`booklib audit --efficiency` command.** The 26.4% "no description" stat and 60% "non-actionable content" stat are implementable checks. `booklib audit` can flag skills that: (a) lack a description in frontmatter, (b) have body sections exceeding a token budget threshold, (c) contain high ratios of non-actionable content (heuristic: paragraphs without imperative verbs, code examples that duplicate rules).
2. **Validates and quantifies the --compact/--progressive approach.** The existing --progressive flag idea (04-01, from OpenDev) is now backed by hard numbers: 39% body reduction with quality improvement. SkillReducer's taxonomy (core rules / supporting examples / reference material) maps cleanly to BookLib's XML tags (`<core_principles>` / `<examples>` / `<resources>`).
3. **Skill authoring guidelines update.** The "less is more" finding compounds with SkillsBench's "2-3 focused modules" finding (04-04) and ETH Zurich's "auto-generated context hurts" finding (04-01). Together they form a compelling three-paper evidence base: write concise, focused skills; don't pad with verbose explanations; trust progressive disclosure.
4. **The adversarial delta debugging technique** is adoptable for `booklib audit --optimize`: given a skill, iteratively remove sentences and test whether retrieval accuracy degrades. Sentences that can be removed without degradation are flagged as "reducible."

**Actionable implementation:**
- Add `booklib audit --efficiency` checks: missing description (26.4% of skills have this!), per-section token counts, non-actionable content ratio
- Update skill authoring docs with SkillReducer's taxonomy: core rules (always inject), supporting examples (inject on demand), reference material (inject only when explicitly requested)
- Cite the 48%/39% compression + 2.8% quality improvement numbers alongside the SkillsBench +16.2pp and ETH Zurich findings in README
- Consider implementing adversarial description compression in `booklib audit --optimize` (future: LLM-powered, but heuristic version first)

**Files:** `lib/engine/auditor.js`, docs/skill-authoring-guide, `bin/cli.js`

## Cross-Cutting Theme

Today's findings coalesce around **the full lifecycle of skill selection and injection**: SkillRouter shows HOW to find the right skill from a large pool (retrieve-and-rerank with full-text access), BGE-reranker-v2-m3-ONNX provides the CONCRETE TOOL to implement cross-encoder reranking in BookLib's Node.js stack, and SkillReducer shows WHAT to do with the skill once selected (compress it for token efficiency, inject progressively).

Together with previous findings, BookLib now has a research-backed blueprint for the complete retrieval pipeline:
1. **Query expansion** (HyDE, dual-level keyword routing — already in IDEAS.md)
2. **First-stage retrieval** (BM25 + vector hybrid with full-text indexing — validated by SkillRouter's body-access finding)
3. **Cross-encoder reranking** (BGE-reranker-v2-m3-ONNX — today's Finding 2 unblocks this)
4. **Progressive injection** (SkillReducer's taxonomy + BookLib's XML tags — today's Finding 3 quantifies the benefit)

The actionable priority chain: (1) integrate BGE-reranker-v2-m3-ONNX as `lib/engine/reranker.js` — lowest effort, highest immediate impact on retrieval quality; (2) add `booklib audit --efficiency` checks using SkillReducer's criteria — low effort, improves skill quality across the ecosystem; (3) adopt SkillRouter's hard negative mining strategy for Spec 2's eval harness — medium effort, foundational for retrieval quality measurement.

## Also Noted (Did Not Pass Full Quality Filter)

- **SKILL0: In-Context Agentic Reinforcement Learning for Skill Internalization** (arxiv:2604.02268, April 2, 2026) — Proposes training skills INTO model weights via progressive curriculum learning, eliminating runtime retrieval. Achieves +9.7% on ALFWorld, +6.6% on Search-QA with <0.5k tokens per step. Interesting counterpoint to BookLib's injection-based approach but not actionable: requires model fine-tuning, which BookLib's architecture (context injection into any LLM) deliberately avoids. Worth monitoring as a long-term competitive threat — if models internalize common skills, the value proposition shifts toward niche/specialized skills.
- **Qwen3 Embedding paper** (arxiv:2506.05176, June 2025) — Describes the Qwen3-Embedding and Qwen3-Reranker model families used by SkillRouter. The 0.6B reranker achieves 73.42 MTEB-Code (higher than BGE-reranker-v2-m3) but lacks ONNX/transformers.js support. Monitor for ONNX conversion.
- **Anthropic 2026 Agentic Coding Trends Report** — Industry report with citable stats (60% AI integration, 97% adoption at Zapier) but no novel research techniques. Useful for positioning materials but not engineering-actionable.
