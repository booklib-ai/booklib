# What We Found Today — April 5, 2026

> The biggest opportunity right now is a bridge between the messy context files everyone actually uses and BookLib's structured skills — and we have the empirical data to prove it.

## The Big Picture

Yesterday's research scan reviewed five papers and sources covering skill ecosystems, security benchmarking, developer adoption patterns, and MCP protocol updates. After validation against source material and feasibility checks, four findings survived — one fully validated and three partially validated. The standout is a clear, data-backed feature that fills a real gap in how developers configure their AI tools.

## What's Worth Building

### The Bridge Nobody Else Has Built

**The problem it solves:** Right now, developers configure their AI coding tools by dropping plain markdown files into their repos — things like AGENTS.md or CLAUDE.md or .cursorrules. It works, but it's unstructured. On the other side, tools like BookLib offer structured skills with tagged sections and searchable content, but adoption is shallow. Developers are stuck choosing between "easy but messy" and "powerful but unfamiliar." Nobody offers a way to move between the two worlds.

**How it works:** Think of it like a translator between two languages. `booklib export` takes a structured skill — with its tagged principles, anti-patterns, and examples — and flattens it into a plain markdown file that any AI tool can read. `booklib import` does the reverse: it reads a plain AGENTS.md file, detects sections by their headings (like "## Principles" or "## Avoid"), and wraps them in BookLib's XML structure. It's not magic — it's heuristic pattern matching on section headings — but it covers the 80% case well.

**What it means for BookLib:** A study of 2,923 GitHub repos by Galster and colleagues at established SE research institutions found that context files dominate how developers configure AI tools, while skills remain barely adopted. This feature turns that finding into an advantage — instead of competing with context files, BookLib becomes the tool that makes them better. Users can import their existing AGENTS.md, get BookLib's structured search and knowledge graph benefits, then export back out for tools that don't support MCP.

**What we'd actually build:** Two new CLI commands using BookLib's existing parser infrastructure. The export side is straightforward template rendering. The import side needs heuristic heading detection — map common markdown patterns to XML tags. Plus a round-trip test to make sure export→import→export produces equivalent output. Estimated at 1–2 days of work.

**Verdict:** ✅ This is the clear P1. No new dependencies, solid empirical backing, and it's a genuine differentiator — no other tool in the ecosystem offers bidirectional conversion.

### Catching Malicious Skills Before They Bite

**The problem it solves:** As skill ecosystems grow, so does the risk of installing something dangerous. A Snyk audit found that out of nearly 4,000 public skills scanned, 534 had critical security issues and 76 contained confirmed malicious payloads. That's roughly one in fifty skills actively trying to do harm. BookLib already has an `audit` command, but it doesn't look for security-specific red flags.

**How it works:** Imagine a building inspector who currently only checks if a house is the right size, but doesn't check the wiring. This adds the wiring check. The new security probes scan skill content for suspicious patterns: URLs that shouldn't be there, shell commands like `exec` or `spawn` hiding in what should be plain instructions, base64-encoded strings that could conceal anything, and prompt injection markers like "ignore previous instructions." Each skill gets a three-level label — safe, caution, or unsafe — borrowed from the SkillTester framework developed at Peking University.

**What it means for BookLib:** When someone runs `booklib fetch` to install a third-party skill, they'll see a clear safety signal before it touches their system. It's the kind of feature that builds trust in the ecosystem, especially as BookLib grows beyond its curated bundle of 22 skills.

**What we'd actually build:** New probe functions in the existing `auditor.js` module — pattern matching against known dangerous patterns. Plus a label display in the fetch command. Half a day to a day of work.

**The catch:** Static analysis catches obvious patterns but can't detect sophisticated attacks. This is a first line of defense, not a guarantee.

**Verdict:** ⚠️ Partially validated — the security probe patterns are solid and adoptable, but the paper's broader comparative utility measurement framework is out of scope for BookLib's architecture. The security piece alone is worth building.

### Making Sure BookLib Speaks Proper MCP

**The problem it solves:** MCP (the Model Context Protocol that lets AI tools talk to knowledge servers like BookLib) released v1.27.1 with a new SDK tiering system. Think of it like accessibility ratings for buildings — there are now official levels of compliance, and tools that meet them get ecosystem trust. BookLib's MCP server works, but hasn't been verified against the official conformance tests.

**How it works:** The MCP project now ships a conformance test suite — essentially a checklist that runs automatically and tells you whether your server meets the spec. It's like plugging your electrical work into a code inspector's test rig. You either pass or you know exactly what to fix. There's also a security patch for URL handling that BookLib should verify it's not affected by.

**What it means for BookLib:** Passing conformance tests is table stakes for being taken seriously in the MCP ecosystem. It's not a differentiator — it's a requirement. And with an MCP Registry expected later this year, being conformance-tested now means BookLib is ready when discovery goes mainstream.

**What we'd actually build:** Add the MCP conformance test suite to BookLib's existing tests, document the current tier level, and review URL handling against the security advisory. Half a day.

**The catch:** The blog posts forecasting Q2 OAuth and Q4 Registry timelines come from third-party analysis, not official MCP sources. Don't plan around those dates.

**Verdict:** ⚠️ Partially validated — the conformance testing is confirmed and practical, but the forward-looking timeline claims are unverified. Do the testing now, plan for the future later.

### Organizing Skills by Category (For Later)

**The problem it solves:** As BookLib's skill library grows, flat search starts to feel like searching a bookstore where every book is shelved randomly. Adding a `taxonomy` field to each skill — like labeling books by genre and sub-genre — would let users filter by category: "show me only the TypeScript patterns" or "just the architecture skills."

**How it works:** Each skill's metadata gets a new optional field — an array like `[languages, typescript, patterns]` that describes where it fits in a knowledge hierarchy. The search command gets a `--category` filter. No changes to the core search engine — just metadata enrichment and filtering on top.

**What it means for BookLib:** At 22 skills, this is nice-to-have. At 500+, it becomes essential. The research behind this (from Shanghai AI Lab's AgentSkillOS project) benchmarks hierarchical search at 200 to 200,000 skills. BookLib isn't there yet, but planting the metadata now — while the Spec 1 frontmatter work is happening anyway — costs almost nothing.

**What we'd actually build:** Add the taxonomy field to the schema, tag 22 existing skills, wire up a `--category` filter. Half a day during existing frontmatter work.

**Verdict:** ⚠️ Partially validated — the full hierarchical search system is overkill at current scale, but the metadata enrichment is practically free and future-proofs the skill library.

Build the bidirectional context file converter.
