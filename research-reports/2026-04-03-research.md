## Finding 1: AgentSkillOS — Capability-Tree Skill Discovery + DAG Orchestration

**Maps to:** Spec 1 (skill discovery), Core architecture (skill routing)
**Impact:** High | **Effort:** Medium
**Paper:** [Organizing, Orchestrating, and Benchmarking Agent Skills at Ecosystem Scale](https://arxiv.org/abs/2603.02176) (March 2, 2026)
**Source type:** Preprint — Shanghai AI Laboratory, ≥4 authors (Hao Li, Chunjiang Mu, Jianhao Chen et al.)
**Credibility:** Multi-author from a major national AI lab; includes benchmark with 30 artifact-rich tasks; LLM-based pairwise evaluation via Bradley-Terry scoring

**Summary:** AgentSkillOS is the first principled framework for skill selection, orchestration, and ecosystem-level management. It has two stages: (1) **Manage Skills** organizes skills into a capability tree via node-level recursive categorization — skills are grouped into semantic clusters at increasing granularity, enabling efficient top-down discovery without scanning the full library. (2) **Solve Tasks** retrieves relevant skills, orchestrates them into DAG-based execution pipelines, and runs them. The benchmark covers 5 categories (data computation, document creation, video, visual design, web interaction) with 30 complex tasks requiring multi-skill composition.

**Why it matters for BookLib:** BookLib's `booklib search` currently uses flat vector search over all skills. AgentSkillOS's capability tree maps directly to a hierarchical skill index — group BookLib's 258+ discoverable skills into a taxonomy tree (language → paradigm → concern), then search top-down. This addresses the scaling problem identified in the "Single-Agent with Skills" paper (2601.04748) where skill selection accuracy drops sharply past a critical library size. The DAG orchestration pattern is also relevant for `booklib context` when multiple skills need to be composed for a complex query.

**Actionable implementation:**
1. Add a `taxonomy` field to skill YAML frontmatter (e.g., `taxonomy: [languages, typescript, patterns]`)
2. In `lib/engine/searcher.js`, implement two-phase search: first narrow to relevant taxonomy subtree, then vector search within that subtree
3. For `booklib scan`, auto-generate taxonomy from existing tags + skill names using a simple clustering heuristic
4. The DAG pattern could inform a future `booklib plan` command that decomposes complex queries into skill-sequenced pipelines

**Competitive edge:** No skill tool currently implements hierarchical skill discovery. skills.sh and agentskills.io use flat search. This would let BookLib scale to 1000+ skills without degrading selection quality.

## Finding 2: SkillTester — Comparative Benchmarking Framework for Skill Utility and Security

**Maps to:** Spec 1 (skill verification), `benchmark/`, `lib/skill-verifier.js`
**Impact:** High | **Effort:** Medium
**Paper:** [SkillTester: Benchmarking Utility and Security of Agent Skills](https://arxiv.org/abs/2603.28815) (March 28, 2026)
**Source type:** Preprint — Peking University + Northwestern Polytechnical University
**Credibility:** Multi-author from two top Chinese universities; novel comparative evaluation methodology; security probe suite included

**Summary:** SkillTester introduces a principled evaluation framework for agent skills with two key innovations: (1) **Comparative utility measurement** — it runs matched baseline (no-skill) and with-skill executions on identical tasks, measuring what the skill *changes* rather than whether it succeeds in isolation. This produces a normalized utility score. (2) **Security probe suite** — separate security evaluation that produces a three-level security status label (safe / caution / unsafe). The framework consolidates rich internal evidence into a small number of interpretable published outputs rather than exposing raw submetrics.

**Why it matters for BookLib:** BookLib already has `booklib audit` for token count warnings, but lacks a way to measure whether a skill actually *helps*. SkillTester's comparative methodology is directly adoptable: run an agent task with and without the skill, measure token usage, completion rate, and output quality diff. This could power:
1. A `booklib bench` command that evaluates skill utility using paired execution (with/without skill)
2. A skill quality badge system for the discovery ecosystem — "verified utility: +15% task completion"
3. The security probe suite maps to the malicious skill detection already in IDEAS.md — SkillTester's three-level labeling (safe/caution/unsafe) is a ready-made UX pattern for `booklib fetch` warnings

**Actionable implementation:**
1. Add `benchmark/skill-tester.js` implementing the comparative utility principle: same task, same model, with/without skill injection
2. Define a minimal eval harness in `benchmark/tasks/` with 5-10 representative coding tasks per skill domain
3. Output utility delta + security label per skill; store in skill metadata for discovery ranking
4. Integrate security probes into `booklib audit` — check for URL patterns, shell commands, encoded content, prompt injection patterns

**Competitive edge:** No skill ecosystem currently offers empirical utility scores. BookLib could be the first to show "this skill makes agents 20% faster at TypeScript tasks" with reproducible evidence. Combined with the 26.1% vulnerability finding from the Agent Skills Survey (below), this positions BookLib as the quality-assured choice.

## Finding 3: Configuring Agentic AI Coding Tools — Skills Adoption Is Still Shallow

**Maps to:** Spec 1 (ecosystem alignment), marketing/positioning
**Impact:** High | **Effort:** Low
**Paper:** [Configuring Agentic AI Coding Tools: An Exploratory Study](https://arxiv.org/abs/2602.14690) (February 16, 2026)
**Source type:** Preprint — 6 authors, Matthias Galster et al. (established SE researchers)
**Credibility:** Empirical study of 2,926 GitHub repositories; systematic analysis of 8 configuration mechanisms across Claude Code, GitHub Copilot, Cursor, Gemini, and Codex

**Summary:** The study identifies 8 configuration mechanisms for AI coding agents and measures their real-world adoption across ~3K repos. Key findings: (1) **Context Files dominate** — AGENTS.md is emerging as an interoperable standard, often the sole mechanism in a repository. (2) **Skills and Subagents are only shallowly adopted** — despite being available across multiple tools, they see minimal real-world usage. (3) Most repos use only 1-2 mechanisms, with Context Files being the default choice. The paper provides a taxonomy of configuration mechanisms: Context Files, Skills, Subagents, Rules, Hooks, MCP Servers, Custom Commands, and Templates.

**Why it matters for BookLib:** This is critical market intelligence. Skills are available but barely used — meaning the ecosystem is in its infancy and BookLib has a window to define best practices before habits solidify. The finding that Context Files dominate suggests BookLib should ensure seamless interop: a skill should be easily extractable to/from AGENTS.md, CLAUDE.md, .cursorrules, etc. The shallow adoption also suggests the barrier is *ease of creation and discovery*, not lack of need — exactly BookLib's value proposition.

**Actionable implementation:**
1. Add `booklib export --format=agents-md` to convert skills into AGENTS.md-compatible format for repos that prefer flat context files
2. Add `booklib import --from=agents-md` to convert existing AGENTS.md/CLAUDE.md content into BookLib skills with XML tags
3. Update positioning materials to cite this study: "Skills adoption is early — BookLib makes it easy"
4. Use the 8-mechanism taxonomy in docs to explain where BookLib fits in the configuration landscape
5. Consider `booklib init` detecting existing context files and offering to convert them

**Competitive edge:** BookLib can position itself as the bridge between the dominant but unstructured Context File pattern and the more powerful but underadopted Skills pattern. No other tool offers bidirectional conversion.

## Finding 4: Agent Skills for LLMs — The Definitive Survey of BookLib's Exact Space

**Maps to:** Core architecture, Spec 1 (ecosystem), marketing/positioning
**Impact:** High | **Effort:** Low
**Paper:** [Agent Skills for Large Language Models: Architecture, Acquisition, Security, and the Path Forward](https://arxiv.org/abs/2602.12430) (February 12, 2026; v3 February 17, 2026)
**Source type:** Preprint survey — multiple authors
**Credibility:** Comprehensive survey with 4 organizing axes; v3 indicates peer feedback iteration; cited by subsequent skill papers (SkillTester, SkillsBench, AgentSkillOS)

**Summary:** This survey formalizes the agent skills paradigm along four axes: (i) **Architectural foundations** — formalizes the SKILL.md specification, progressive context loading, and the complementary roles of skills and MCP; (ii) **Skill acquisition** — covers RL with skill libraries, autonomous skill discovery (SEAgent), and compositional skill synthesis; (iii) **Deployment at scale** — CUA stack, GUI grounding, benchmark progress on OSWorld and SWE-bench; (iv) **Security** — empirical analysis showing **26.1% of community-contributed skills contain vulnerabilities**. The paper explicitly describes progressive disclosure as a core architectural principle.

**Why it matters for BookLib:** This survey essentially validates BookLib's entire design thesis in academic literature. The SKILL.md specification it formalizes is the format BookLib pioneered. The progressive disclosure principle maps to BookLib's `--compact` and `--progressive` modes. The 26.1% vulnerability stat reinforces the urgency of `booklib audit`'s security checks (already flagged in IDEAS.md but now with a citable number). Most importantly, being cited as the implementation of the patterns this survey describes would be enormous for credibility.

**Actionable implementation:**
1. Ensure BookLib's SKILL.md format aligns with the specification described in this survey — verify field names, required sections, and metadata schema match
2. Add a `RESEARCH.md` section mapping BookLib features to each of the survey's 4 axes
3. Cite the 26.1% vulnerability stat in `booklib fetch` warning messages and security documentation
4. Reach out to the survey authors — BookLib may qualify for inclusion in a future revision as a reference implementation
5. Use the survey's terminology ("progressive context loading", "compositional skill synthesis") in BookLib's technical docs for discoverability

**Competitive edge:** Being the open-source implementation that a definitive survey describes is the strongest possible positioning. No competitor can claim this alignment.

## Finding 5: MCP v1.27 SDK Tiering System (SEP-1730) — Compliance Target for BookLib's MCP Server

**Maps to:** Spec 1 (MCP server), `lib/mcp/server.js`
**Impact:** Medium | **Effort:** Low
**Source:** [MCP v1.27.1 Release](https://modelcontextprotocol.io/specification/2025-11-25/changelog) + [MCP Ecosystem in 2026](https://www.contextstudios.ai/blog/mcp-ecosystem-in-2026-what-the-v127-release-actually-tells-us) (February 24, 2026)
**Source type:** Official MCP specification documentation
**Credibility:** First-party protocol specification; TypeScript SDK has 34,700+ dependent projects

**Summary:** MCP v1.27.1 introduced the **SDK Tiering System (SEP-1730)** with clear requirements for feature support and maintenance commitments at each tier. The release focuses on correctness: auth/pre-registration conformance testing, a command injection vulnerability fix in URL handling, and patched silently-swallowed transport errors. Separately, the MCP roadmap confirms: enterprise OAuth 2.1 + PKCE auth scheduled for Q2 2026, and an official MCP Registry with security audits scheduled for Q4 2026. The TypeScript SDK is the reference implementation with the most active conformance work.

**Why it matters for BookLib:** BookLib's MCP server (`lib/mcp/server.js`) should target a specific SDK tier to ensure ecosystem compatibility and signal quality. The conformance testing infrastructure means BookLib can now *verify* its MCP implementation meets spec requirements — not just claim compliance. The Q4 2026 MCP Registry is a major distribution channel opportunity; being a conformant, tiered server from day one will be an advantage when the registry launches with security audits.

**Actionable implementation:**
1. Review SEP-1730 tier requirements and identify which tier BookLib's MCP server currently meets
2. Add MCP conformance tests to BookLib's test suite using the official testing infrastructure
3. Fix any URL handling patterns that might be affected by the command injection vulnerability patch
4. Plan OAuth 2.1 support for when enterprise auth ships in Q2 2026 — this unlocks corporate deployment
5. Prepare a server registration package for the Q4 2026 MCP Registry launch

**Competitive edge:** Early SDK tier compliance and conformance testing positions BookLib for automatic inclusion in the official MCP Registry. Competitors without conformance will face a gatekeeping barrier.

## Cross-Cutting Theme

Today's findings tell a coherent story about the **maturation of the agent skills ecosystem**. The space has moved from "does this concept work?" (answered by previous reports' ETH Zurich and Lulla et al. studies) to "how do we organize, evaluate, and secure skills at scale?" AgentSkillOS addresses discovery scaling, SkillTester addresses quality measurement, the Galster study maps real adoption patterns, and the comprehensive survey formalizes the architecture. BookLib is uniquely positioned because it already implements the patterns these papers describe — the gap is now about **ecosystem tooling** (import/export, benchmarking, conformance) rather than core retrieval.
