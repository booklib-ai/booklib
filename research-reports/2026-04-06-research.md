## Finding 1: SkillFlow v2 — 4-Stage Progressive Retrieval Pipeline for 36K SKILL.md Corpus

**Maps to:** Spec 2 (retrieval quality), `lib/engine/searcher.js`, `lib/engine/reranker.js`
**Impact:** High | **Effort:** Medium

**Paper:** [SkillFlow: Scalable and Efficient Agent Skill Retrieval System](https://arxiv.org/abs/2504.06188) (v2: March 27, 2026)
**Source type:** Preprint — 3 authors (Fangzhou Li, Pagkratios Tagkopoulos, Ilias Tagkopoulos), UC Davis + USDA/NSF AI Institute
**Credibility:** 4/5 — Open-source implementation ([github.com/IBPA/skill-flow](https://github.com/IBPA/skill-flow)), evaluated on SkillsBench (87 tasks, 229 skills) and Terminal-Bench (89 tasks), v2 is a major revision with new title and architecture. UC Davis is a reputable institution.

**Summary:** SkillFlow is the first multi-stage retrieval pipeline designed specifically for SKILL.md discovery, framing skill acquisition as an information retrieval problem over a corpus of ~36K community-contributed skill definitions indexed from GitHub. Unlike SkillRouter's 2-stage approach (already reported 04-05), SkillFlow uses a **4-stage progressive narrowing pipeline**: (1) dense retrieval over the full corpus to get top-N candidates, (2) shallow cross-encoder reranking for initial quality filtering, (3) deep cross-encoder reranking for fine-grained scoring, and (4) LLM-based final selection that evaluates the top candidates with full task context. Each stage balances recall and precision — wide net first, progressively narrower. A critical design choice: SkillFlow operates over *structured* skill definitions that mix natural-language descriptions with YAML metadata and embedded code, not homogeneous text passages.

**How it differs from SkillRouter (04-05):** SkillRouter is a 2-stage system (0.6B embedder → 0.6B reranker = 1.2B total) focused on the *training methodology* (listwise loss, hard negative mining, false negative filtering). SkillFlow is a *pipeline architecture* paper — it's agnostic to the specific models used at each stage and focuses on the optimal number and ordering of stages. The key novel contribution is the LLM-as-final-selector stage: after two rounds of cross-encoder reranking, an LLM evaluates the top candidates with full task context and makes the final selection. This addresses the problem that cross-encoders score query-document relevance but can't reason about task intent.

**Why it matters for BookLib:** SkillFlow's architecture maps directly onto the retrieval pipeline BookLib is building in Spec 2, but adds a dimension not yet considered: a final LLM-based selection stage. The A/B test findings (04-03) already showed that BookLib's Haiku-based synthesizer adds significant value (+3 to +5 over raw retrieval). SkillFlow formalizes this pattern — the synthesizer IS the LLM selection stage. Specific implications:

1. **BookLib's pipeline already approximates SkillFlow.** Current: BM25+vector (stage 1-2) → RRF fusion → Haiku synthesizer (stage 4). Adding BGE-reranker-v2-m3-ONNX (from 04-05) as stage 3 would complete the 4-stage architecture.
2. **The two-round reranking distinction matters.** SkillFlow uses a *shallow* reranker (fast, lower quality) then a *deep* reranker (slower, higher quality). For BookLib, this maps to: shallow = the early-exit SEE approach (04-02), deep = full BGE-reranker-v2-m3. This justifies keeping both reranking approaches rather than choosing one.
3. **The 36K SKILL.md corpus is a validation dataset.** SkillFlow indexed 36K skills from GitHub — BookLib could use the same corpus to benchmark its retrieval pipeline against SkillFlow's published numbers.

**Actionable implementation:**
- Formalize BookLib's pipeline as 4-stage in `lib/engine/searcher.js`: (1) BM25+vector hybrid → (2) SEE early-exit reranker (shallow) → (3) BGE-reranker-v2-m3 (deep) → (4) Haiku synthesizer (LLM selection)
- Benchmark against SkillFlow's results on SkillsBench to measure BookLib's competitive retrieval quality
- Consider indexing SkillFlow's 36K skill corpus alongside BookLib's 280 skills for stress-testing at scale

**Files:** `lib/engine/searcher.js`, `lib/engine/reranker.js`, `benchmark/`

## Finding 2: SoK: Agentic Skills — Seven Design Patterns and the ClawHavoc Campaign

**Maps to:** Core architecture, Spec 1 (ecosystem + security), marketing/positioning
**Impact:** High | **Effort:** Low

**Paper:** [SoK: Agentic Skills — Beyond Tool Use in LLM Agents](https://arxiv.org/abs/2602.20867) (February 24, 2026)
**Source type:** Preprint — Systematization of Knowledge (SoK) paper
**Credibility:** 4/5 — SoK papers undergo rigorous systematization methodology; covers full skill lifecycle with formal taxonomies; includes the ClawHavoc case study grounded in real-world incident data. Multi-author.

**Summary:** This SoK paper maps the skill layer across the full lifecycle — discovery, practice, distillation, storage, composition, evaluation, and update — and introduces two complementary taxonomies. The first is a **system-level set of seven design patterns** capturing how skills are packaged and executed in practice: (1) metadata-driven progressive disclosure, (2) executable code skills, (3) self-evolving skill libraries, (4) marketplace distribution, (5) skill composition/chaining, (6) trust-tiered execution, and (7) template-based skill scaffolding. The second taxonomy covers evaluation approaches anchored by SkillsBench evidence. The paper's most striking contribution is the **ClawHavoc case study** — a real-world incident in which nearly 1,200 malicious skills infiltrated a major agent marketplace, exfiltrating API keys, cryptocurrency wallets, and browser credentials at scale.

**How it differs from the "Agent Skills for LLMs" survey (04-03):** The survey (2602.12430) is a broad overview organized along 4 axes (architecture, acquisition, deployment, security). This SoK is a formal systematization with *prescriptive* design patterns and a real-world security incident analysis. The 7 design patterns are new — no previous report captured these. The ClawHavoc campaign (1,200 malicious skills) is a much larger incident than the 341 malicious skills noted in the 03-31 security alert or the 26.1% vulnerability rate from the survey.

**Why it matters for BookLib:**

1. **Design pattern #1 (metadata-driven progressive disclosure) IS BookLib's architecture.** The SoK formally names and validates the pattern BookLib uses: skill metadata → summary → core principles → full body. Being aligned with a formally named design pattern from a SoK paper strengthens positioning.
2. **Design pattern #6 (trust-tiered execution) maps to `booklib fetch` security.** The SoK recommends a tiered trust model: built-in skills (trusted) → curated marketplace skills (verified) → community skills (sandboxed). BookLib's existing three-level labeling (safe/caution/unsafe) from SkillTester aligns perfectly. The SoK provides the formal framework for why this tiering matters.
3. **ClawHavoc campaign (1,200 malicious skills) is a citable incident** for BookLib's security documentation and marketing. The scale (1,200 skills, exfiltrating API keys + crypto wallets) is dramatic and concrete — much more compelling than abstract vulnerability percentages.
4. **The 7-pattern taxonomy is a marketing/docs asset.** BookLib can position itself by mapping its features to each of the 7 patterns and showing which ones it implements vs. competitors.

**Actionable implementation:**
- Create a section in RESEARCH.md mapping BookLib features to each of the 7 design patterns
- Update security documentation to cite ClawHavoc (1,200 malicious skills, API key + wallet exfiltration)
- Implement trust-tiered execution in `lib/skill-fetcher.js`: built-in skills skip verification, marketplace skills get Tier 2 context check, community skills get full Tier 1 + Tier 2 scanning
- Use the lifecycle taxonomy (discovery → practice → distillation → storage → composition → evaluation → update) to organize BookLib's feature roadmap

**Files:** RESEARCH.md, docs/, `lib/skill-fetcher.js`, `lib/skill-verifier.js`

## Finding 3: Skill-Inject Benchmark — 80% Attack Success Rate on Frontier Models via Skill Files

**Maps to:** Spec 1 (security), `lib/skill-verifier.js`, `benchmark/security/`
**Impact:** High | **Effort:** Medium

**Paper:** [Skill-Inject: Measuring Agent Vulnerability to Skill File Attacks](https://arxiv.org/abs/2602.20156) (February 2026, v2)
**Source type:** Preprint — multi-author
**Credibility:** 4/5 — Dedicated benchmark with 202 injection-task pairs, tested across frontier models, quantitative attack success metrics, includes both obvious and subtle attack variants. Accompanied by SkillJect (2602.14211), a companion automated attack generation framework from Peking University with open code.

**Summary:** Skill-Inject is the first dedicated benchmark measuring LLM agent vulnerability to skill file injection attacks. It contains 202 injection-task pairs with attacks ranging from obviously malicious injections to subtle, context-dependent attacks hidden in otherwise legitimate instructions. Results: **up to 80% attack success rate** with frontier models, with agents often executing extremely harmful instructions including data exfiltration, destructive file operations, and ransomware-like behavior. The companion paper SkillJect (arxiv:2602.14211) automates the generation of stealthy skill-based prompt injections using a closed-loop approach: an Attack Agent synthesizes injection payloads, a Code Agent executes tasks with the injected skill, and an Evaluate Agent checks whether the malicious behavior occurred. The payload hiding strategy conceals adversarial operations in auxiliary scripts while injecting inducement prompts to trigger tool execution.

**How it differs from previous security findings:** The 03-31 security alert noted 341 malicious skills. The 04-03 survey cited 26.1% vulnerability rates. The 04-04 "Malicious Or Not" paper addressed false positive reduction. Skill-Inject is the first to provide **quantitative attack success rates against specific models** and a **reusable benchmark** for testing defenses. The 80% figure is significantly more alarming than abstract vulnerability percentages because it measures actual successful exploitation, not just the presence of suspicious patterns.

**Why it matters for BookLib:**

1. **The 80% attack success rate is the strongest argument for BookLib's security features.** When marketing `booklib audit` and `booklib fetch` security scanning, this is the "why you need this" number.
2. **The 202-pair benchmark is adoptable for testing.** BookLib's security scanning (planned in `lib/skill-verifier.js`) can be evaluated against Skill-Inject's benchmark to measure detection rates. This gives a concrete answer to "how good is our security?"
3. **SkillJect's attack taxonomy informs defense.** The paper categorizes attacks into: (a) data exfiltration via auxiliary scripts, (b) destructive file operations, (c) subtle behavioral modifications. Each category needs different detection heuristics in `booklib audit`:
   - Exfiltration: scan for `fetch`, `http`, `curl` in referenced scripts
   - Destruction: scan for `rm -rf`, `fs.unlink`, `exec` in skill content
   - Behavioral: harder — requires semantic analysis of skill instructions
4. **The "subtle attack" variant is the key challenge.** Skill-Inject shows that attacks hidden in context-dependent instructions are much harder to detect than obvious ones. This validates the need for the repo-context approach from "Malicious Or Not" (04-04) — content scanning alone isn't enough.

**Actionable implementation:**
- Add `benchmark/security/` directory with Skill-Inject's 202 test pairs adapted for BookLib's format
- Implement detection heuristics in `lib/skill-verifier.js` targeting SkillJect's three attack categories
- Benchmark BookLib's security scanner detection rate against Skill-Inject and report as a metric
- Cite the 80% figure alongside ClawHavoc (Finding 2) in security marketing materials
- Consider a `booklib audit --security` command that runs the full Skill-Inject-inspired test suite

**Files:** `lib/skill-verifier.js`, `benchmark/security/`, docs/security

## Finding 4: EvoSkill — Failure-Driven Skill Evolution as Competitive Benchmark

**Maps to:** Core (competitive landscape), Spec 2 (eval harness), quality validation
**Impact:** Medium | **Effort:** Low (insights, not code)

**Paper:** [EvoSkill: Automated Skill Discovery for Multi-Agent Systems](https://arxiv.org/abs/2603.02766) (March 3, 2026)
**Source type:** Preprint — 5 authors (Salaheddin Alzubi, Noah Provenzano, Jaydon Bingham, Weiyuan Chen, Tu Vu), Sentient AGI
**Credibility:** 3.5/5 — Open-source implementation ([github.com/sentient-agi/EvoSkill](https://github.com/sentient-agi/EvoSkill)), evaluated on OfficeQA and SealQA benchmarks with quantitative results. Sentient AGI is a newer lab but the results are reproducible.

**Summary:** EvoSkill is a self-evolving framework that automatically discovers and refines agent skills through iterative failure analysis. Rather than optimizing prompts or code directly, it iteratively proposes new skills or edits existing ones based on execution failure traces, then validates improvements on a held-out set. A Pareto frontier governs skill selection — only skills that improve validation performance are retained. Results: **+7.3% on OfficeQA with Claude Code + Opus 4.5** (60.6% → 67.9%), **+12.1% on SealQA** (26.6% → 38.7%). Most notably, a skill evolved on SealQA **transfers zero-shot to BrowseComp** with a +5.3% improvement — demonstrating that evolved skills generalize beyond their training task.

**How it differs from previously reported auto-generation findings:** The SkillsBench study (04-04) showed that self-generated skills provide "zero average benefit." SKILL0 (noted in 04-05) proposed internalizing skills into model weights. EvoSkill is different in two key ways: (1) it evolves skills through *failure analysis*, not one-shot generation — the iterative refinement loop is what makes it work where one-shot generation fails; (2) the skills it produces are structured, reusable SKILL.md-format artifacts that persist across sessions, not in-context or in-weight modifications.

**Why it matters for BookLib:**

1. **EvoSkill's +7.3% with auto-evolved skills vs. SkillsBench's 0% with self-generated skills tells a nuanced story.** The difference is iterative refinement vs. one-shot generation. This validates BookLib's curation approach (curated > one-shot generated) while opening a future direction: semi-automated skill refinement using EvoSkill's failure-driven loop. A `booklib evolve` command that takes a skill + benchmark failures and proposes improvements could combine the best of both worlds.
2. **The zero-shot transfer finding validates skill portability.** If auto-evolved skills transfer across tasks, then BookLib's curated skills (which are more general by design) should transfer even better. This is a useful data point for marketing.
3. **EvoSkill's Pareto frontier for skill selection** is adoptable for BookLib's `booklib audit` quality gate. Instead of just checking token counts, test whether a skill improves performance on a held-out task set. Skills that don't improve — or actively hurt — should be flagged.
4. **Competitive landscape insight.** If EvoSkill matures, auto-evolved skills could eventually close the quality gap with curated skills. BookLib's moat then shifts from "curated > auto-generated" to "curated + search + progressive disclosure" — the delivery mechanism rather than just content quality.

**Actionable implementation:**
- Add EvoSkill's benchmarks (OfficeQA, SealQA) as optional eval targets for `benchmark/`
- Consider a future `booklib evolve` command that takes skill + failure traces and suggests refinements
- Update competitive landscape docs: auto-evolved skills can reach +7% with iterative refinement (EvoSkill) but 0% with one-shot generation (SkillsBench) — curated skills remain the gold standard
- Use the zero-shot transfer finding in marketing: "Skills built for one task improve others — that's the power of curated procedural knowledge"

**Files:** `benchmark/`, docs/competitive-landscape, README

## Cross-Cutting Theme

Today's findings reveal **two complementary maturation tracks in the skill ecosystem**: the retrieval track and the security track, now both backed by substantial benchmarks.

On the **retrieval side**, SkillFlow v2 provides the first reference architecture for multi-stage skill retrieval at scale (36K skills), and it validates the pipeline BookLib is already building: BM25+vector → reranking → LLM selection. BookLib's A/B test findings (04-03) showed that the Haiku synthesizer adds +3 to +5 over raw retrieval — SkillFlow formalizes this as the "LLM selection stage" of a multi-stage pipeline. Combined with SkillRouter (04-05) and BGE-reranker-v2-m3-ONNX (04-05), BookLib now has two reference architectures plus a concrete model to draw from.

On the **security side**, the picture is increasingly urgent. The progression across reports: 341 malicious skills (03-31) → 26.1% vulnerability rate (04-03) → 46.8% false positive rate in naive scanning (04-04) → now 1,200 malicious skills in a single campaign (ClawHavoc, today) and 80% attack success rate (Skill-Inject, today). Each new finding makes the case for security features stronger. The actionable chain: (1) implement content scanning with SkillJect's attack taxonomy, (2) reduce false positives with repo-context from "Malicious Or Not" (04-04), (3) benchmark detection rates against Skill-Inject's 202 pairs, (4) implement trust-tiered execution from the SoK's design pattern #6.

The **competitive insight from EvoSkill** adds nuance to BookLib's positioning: auto-evolved skills CAN work (+7.3%) if iteratively refined from failures, but they still require significant compute and don't match the breadth of curated expert knowledge. BookLib's value proposition evolves from "curated > auto-generated" to "curated + intelligent retrieval + security = the complete skill delivery platform."

The pre-entity extraction pattern is interesting for Spec 3 but falls outside the recency window.
