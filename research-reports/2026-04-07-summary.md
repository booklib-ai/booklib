## What's Worth Building

### A Better Brain for Search

**The problem it solves:** Right now, BookLib uses all-MiniLM-L6-v2 to understand what you're searching for — a small, fast model that gets the job done but doesn't deeply understand code. When you search for something nuanced like "how to handle Stripe webhook retries," the current model sometimes misses relevant skills because it doesn't grasp code-specific concepts as well as it could.

**How it works:** Alibaba's Qwen team released Qwen3-Embedding-0.6B — think of it as upgrading from a general-purpose dictionary to one written specifically for programmers. It's been trained on 119 languages and excels at understanding code retrieval queries. The model is already packaged in ONNX format (the same format BookLib uses today) and confirmed working with Transformers.js, so the integration path is clean.

**What it means for BookLib:** Users would notice more relevant search results, especially for technical queries. When BookLib recommends skills to inject into your AI's context, better embeddings mean better recommendations — the right knowledge surfaces at the right time. This strengthens BookLib's core value proposition of "curated + high-quality retrieval."

**What we'd actually build:** A model selection option in `embedding-provider.js` — the current model stays as the fast default, and users opt into Qwen3 via `booklib config set model qwen3-0.6b`. First use triggers a lazy download with a progress bar, and switching models requires a re-index.

**The catch:** At 0.6B parameters (~1.2GB), this model is 27x larger than the current one (~90MB). Cold start and per-query latency need real benchmarking before we commit. It also depends on the Transformers.js v4 upgrade we're already tracking.

**Verdict:** ✅ High-impact upgrade — pursue after the Transformers.js v4 migration.

### BookLib Needs to Show Up Where Agents Shop

**The problem it solves:** The agent skills ecosystem has exploded. Anthropic published a spec, OpenAI adopted it, VS Code integrated it, and marketplaces like SkillsMP and skills.sh have emerged. If BookLib's skill format doesn't align with the agentskills.io specification, and if our 22 curated skills aren't listed on these marketplaces, we're invisible to the growing number of developers browsing for quality skills.

**How it works:** The Agent Skills specification defines a standard format for how skills describe themselves — metadata fields, capability declarations, and discovery endpoints. Think of it like making sure your restaurant is listed on Google Maps and Yelp, not just having a sign on the door. You publish a `.well-known/agent-skills.json` file (similar to how websites publish `robots.txt`) so that tools and marketplaces can automatically find and index your skills.

**What it means for BookLib:** Free discoverability across VS Code Copilot, Claude Code, Cursor, and Codex/ChatGPT — all of which now support the spec. BookLib's curated, expert-written skills stand out in a marketplace where quality is rare. This is marketing that costs zero ongoing effort.

**What we'd actually build:** An audit of our 22 bundled skills against the spec's field requirements, a `booklib build-wellknown` enhancement that generates `agent-skills.json` alongside the existing MCP server card, and submissions to SkillsMP and skills.sh.

**The catch:** The ecosystem size numbers floating around are inflated — one source claimed 425K skills on SkillsMP when independent reviews put it closer to 66K. The growth narrative is real, but take specific numbers with a grain of salt.

**Verdict:** ⚠️ Low effort, high reward — but verify format compliance before submitting. Ecosystem numbers are unreliable.

### Graphs That Grow Instead of Being Built

**The problem it solves:** When BookLib eventually builds its knowledge graph (Spec 3), there's a design question: do you construct the entire graph upfront every time content changes, or do you grow it incrementally? Upfront construction is simpler to reason about but gets painfully slow as the knowledge base grows.

**How it works:** A paper from a Tsinghua-adjacent research group introduces GraphAnchor, which builds graphs the way you'd build a puzzle — adding pieces as you encounter them rather than dumping all pieces on the table first. Each time new content arrives, only the new entities and relationships get added. Imagine a library that updates its card catalog by just filing new cards for new books, instead of re-cataloging everything from scratch.

**What it means for BookLib:** This validates the append-only JSONL edge storage that Spec 3 already plans. When you run `booklib capture`, the graph should expand incrementally — entity extraction on new content only, edges appended, never reconstructed. This keeps the capture command fast even as your knowledge base grows.

**What we'd actually build:** Nothing right now. This is a design reference for when Spec 3 begins. The paper's runtime graph expansion (which requires an LLM in the loop) is not applicable — BookLib needs search under 500ms, not multi-hop question answering.

**Verdict:** ⚠️ Sound design principle, but the paper's runtime pattern doesn't apply. File under "Spec 3 reference material."

### A Cheat Sheet for Building Better Benchmarks

**The problem it solves:** BookLib needs an eval harness (Spec 2) to measure whether search improvements actually help users. But designing good retrieval benchmarks for code is tricky — what metrics matter? What chunk formats work best? How do you compare different retrieval strategies fairly?

**How it works:** A comprehensive survey of Retrieval-Augmented Code Generation catalogues the current landscape of how researchers evaluate code retrieval systems. It covers generation strategies, retrieval modalities, and evaluation protocols — essentially a "state of the art" map for anyone building code retrieval benchmarks.

**What it means for BookLib:** Instead of inventing our eval methodology from scratch, we can stand on the shoulders of this systematic review. The survey's comparison of chunk formats and context structures directly informs how BookLib should structure skill chunks for optimal retrieval in code generation tasks.

**What we'd actually build:** No code — this is reading homework. The survey's evaluation methodology should be referenced when designing the Spec 2 benchmark harness.

**Verdict:** ✅ Solid design input. Read it before designing the eval harness.
