## What's Worth Building

### Your Curated Skills Are Doing Exactly What the Research Says They Should

**The problem it solves:** Every AI tool vendor promises that "just give the agent more context and it'll be smarter." But that's not what the data shows. Dumping auto-generated context on an agent actually makes it slower and more expensive. The question is: does *structured, human-written* context actually help? Now we have hard numbers.

**How it works:** Researchers studied 10 real GitHub repositories and 124 pull requests, comparing repos that had human-authored AGENTS.md files against those that didn't. Think of it like comparing a team that writes good onboarding docs versus one that just throws new hires into the deep end. The teams with docs got their AI agents to finish tasks 28.64% faster while using 16.58% fewer tokens — that's real money saved on every API call.

**What it means for BookLib:** This is the second paper validating BookLib's entire philosophy. The first (from ETH Zurich) showed that auto-generated context *hurts* — 20%+ cost increase with worse results. This new one shows that curated context *helps* — 28% faster with fewer tokens. Together, they tell a compelling story: BookLib's structured XML format, which forces you to curate rather than dump, is exactly what the research says works.

**What we'd actually build:** No new features needed — this is about documentation and positioning. The two-paper narrative ("curated = 28% faster; auto-generated = 20% more expensive") should go into the README, skill authoring guidelines, and `booklib init` output. This merges into the existing "Research-backed skill content guidelines" work that's already a Hot P1.

**Verdict:** ✅ Reinforces the highest-priority item on the backlog. Zero additional effort — just stronger evidence for work already planned.

### Auto-Building Knowledge Graphs Without an LLM

**The problem it solves:** Right now, BookLib's knowledge graph relies on manual capture — you tell your agent "remember this" and it stores it. But what if you wanted to automatically extract structured knowledge from documents? The usual approach requires sending everything through an LLM, which is slow and expensive.

**How it works:** Imagine diagramming a sentence the way you did in school — identifying the subject, verb, and object. This paper uses that same idea (dependency parsing) to automatically build knowledge graphs from text, achieving 94% of the quality you'd get from an LLM but at a fraction of the cost. They combine this with a multi-layered search that looks at entities, relationships, and text chunks separately, then merges the results.

**What it means for BookLib:** Honestly, not much right now. BookLib deliberately chose manual capture over auto-extraction — the whole point is that curated knowledge beats auto-generated noise. The project already has reciprocal rank fusion implemented. But this is a useful data point filed away for the future: if BookLib ever grows to a scale where manual capture can't keep up, dependency parsing is the way to auto-extract without breaking the bank on LLM calls.

**The catch:** No open-source code available, tested only on enterprise code migration (very different from programming skills), and the multi-granular embedding approach would triple BookLib's vector index complexity — overkill for a 22-skill corpus.

**Verdict:** ⚠️ Sound research, but a future reference rather than something to build today. Filed as P3.

### Better Contextual Embeddings (In Theory)

**The problem it solves:** When you chop a document into chunks and embed each one separately, you lose the big picture. A paragraph about "it" referring to authentication patterns just becomes a paragraph about "it" — the embedding doesn't know what "it" means without the surrounding context.

**How it works:** Researchers from EPFL built a benchmark called ConTEB showing that even the best embedding models fail when meaning depends on surrounding context. Their solution, InSeNT, trains models to embed chunks while "remembering" the full document — like reading an entire chapter before summarizing each paragraph. They released open-source code and pre-trained models.

**What it means for BookLib:** The concept validates something already on the backlog — late chunking, where you embed the whole skill file first and then split it up. But the specific models they released are 12x larger than what BookLib uses (395M vs 33M parameters), way too heavy for CPU-only search under 500ms. And InSeNT requires model fine-tuning infrastructure that BookLib doesn't have. The real takeaway is that the ConTEB benchmark could be useful for testing BookLib's own late chunking implementation once it's built.

**The catch:** Models too large for CPU inference, requires training infrastructure BookLib doesn't have, and the core late chunking insight is already captured in the backlog.

**Verdict:** ⚠️ Good validation that contextual embeddings matter, but nothing new to build. Strengthens the existing late chunking backlog item.
