# What We Found Today — April 4, 2026

> The biggest insight: a study of nearly 3,000 repos proves that flat context files dominate AI tool configuration — and BookLib is perfectly positioned to bridge the gap between those files and structured skills.

## The Big Picture

Today's scan reviewed 5 papers and specs from the AI agent skills landscape. After validation against actual source material and BookLib's existing codebase, 1 finding was fully validated and 3 were partially validated — giving us 4 actionable ideas, one of which stands out as a clear priority.

## What's Worth Building

### The Bridge Nobody's Built Yet

**The problem it solves:** Right now, there are two camps in the AI coding tool world. Camp A writes flat markdown files — AGENTS.md, CLAUDE.md, .cursorrules — because they're simple and every tool understands them. Camp B (that's us) uses structured skills with XML tags because structure means better search and better results. The problem? Almost nobody converts between the two. If you've spent hours writing a great CLAUDE.md, there's no easy way to upgrade it into a BookLib skill. And if you've built BookLib skills, there's no way to share them with teammates who use simpler tools.

**How it works:** Think of it like a universal translator. `booklib export` takes a structured skill and flattens it into whatever format you need — plain markdown for AGENTS.md, a different flavor for .cursorrules. `booklib import` does the reverse: it reads your existing flat file, detects sections based on headings (like "## Principles" or "## Avoid"), and wraps them in the right XML tags to create a proper BookLib skill. It's not magic — it uses simple heuristic pattern matching — but it covers the 80% case.

**What it means for BookLib:** This is a genuine gap in the ecosystem. A study by Galster and five other established software engineering researchers analyzed 2,923 GitHub repositories and found that context files absolutely dominate how people configure their AI tools, while skills remain shallowly adopted. BookLib becomes the on-ramp: "Already have a CLAUDE.md? Import it. Want to share with non-BookLib users? Export it." That's a powerful adoption story.

**What we'd actually build:** Two new CLI commands — export and import — plus a new `lib/importer.js` module. The export side mostly reuses existing parser infrastructure. The import side needs heuristic heading detection, which is straightforward text processing. Estimated at 1–2 days of work.

**Verdict:** ✅ P1 — Build this. It directly supports Spec 1 and no other tool offers it.

### Smarter Security Scanning for Third-Party Skills

**The problem it solves:** When someone downloads a skill from the internet, how do you know it's safe? A Snyk audit of nearly 4,000 public skills found that 534 had critical security issues and 76 contained confirmed malicious payloads. That's roughly 1 in 7 skills with something wrong and about 1 in 50 actively trying to cause harm.

**How it works:** Imagine a TSA scanner for skill files. Before a skill gets installed, BookLib would scan it for red flags: suspicious URLs, shell commands that could execute arbitrary code (things like `exec` or `spawn`), encoded content that might be hiding something, and prompt injection patterns (like "ignore previous instructions"). Each skill gets a simple traffic-light label — safe, caution, or unsafe — so you know what you're dealing with before it touches your project.

**What it means for BookLib:** Security is table stakes for any tool that installs third-party code. This makes `booklib fetch` and `booklib audit` meaningfully safer. The three-level labeling is borrowed from SkillTester, a benchmarking framework out of Peking University, and it's an intuitive UX pattern that communicates risk without overwhelming users.

**What we'd actually build:** New security probes added to the existing `auditor.js`, plus a security label displayed during `booklib fetch`. About half a day to a full day of work.

**The catch:** The security probes are static analysis — they catch obvious patterns but won't detect sophisticated attacks. They're a first line of defense, not a complete solution.

**Verdict:** ⚠️ P2 — Solid incremental improvement. Add to the existing audit command work already planned.

### Playing Nice with the MCP Ecosystem

**The problem it solves:** The Model Context Protocol just shipped v1.27.1 with a new SDK Tiering System. Think of it like a certification program — MCP servers can now prove they meet certain compliance levels through standardized conformance tests. If BookLib doesn't run these tests, we're flying blind on whether our MCP server actually meets the spec.

**How it works:** The MCP project published an official conformance test suite. You run it against your server and it tells you what tier you qualify for and where you fall short. It's like running a linter, but for protocol compliance. The v1.27.1 release also patched a command injection vulnerability in URL handling that BookLib should review.

**What it means for BookLib:** This is compliance work, not innovation — but it matters. When the expected MCP Registry launches (likely later this year), servers that can demonstrate tier compliance will have an advantage in discovery and trust. Getting ahead of this now is cheap insurance.

**What we'd actually build:** Add conformance tests to BookLib's test suite, document the current tier level, and review URL handling against the security advisory. About half a day.

**The catch:** Some of the timeline predictions (OAuth 2.1 in Q2, MCP Registry in Q4) come from a third-party blog, not official MCP sources. Don't over-invest in unconfirmed roadmap items.

**Verdict:** ⚠️ P2 — Low effort, good ecosystem hygiene. Fold into the existing MCP Server Cards backlog item.

### Taxonomy Tags for Future-Proofing

**The problem it solves:** As BookLib's skill library grows, users will want to browse and filter by category — "show me all TypeScript skills" or "just the security-related ones." Right now there's no structured way to do that.

**How it works:** Each skill gets an optional `taxonomy` field in its YAML frontmatter — a simple array like `[languages, typescript, patterns]`. It's just metadata, like adding genre tags to songs in your music library. No changes to the search engine required.

**What it means for BookLib:** At 22 bundled skills, this is nice-to-have. At 500+ skills, it becomes essential. Adding the metadata now while doing other frontmatter work (Spec 1.1) is nearly free. The `booklib search --category=...` filter it enables is a natural UX improvement.

**What we'd actually build:** Add the field to the schema, tag the 22 existing skills, and wire up a category filter flag. Half a day, done alongside existing Spec 1.1 work.

**Verdict:** ⚠️ P3 — Low effort, plant the seed now. Don't build hierarchical search until the corpus justifies it.

## If I Had to Pick One Thing

Build the bidirectional context file converter. It's the only finding today backed by hard empirical data (2,923 repos), it fills a gap no other tool addresses, and it turns BookLib into the natural bridge between the "simple flat file" world and the "structured skills" world. Every CLAUDE.md in the wild becomes a potential BookLib user. That's not incremental — that's an adoption multiplier.
