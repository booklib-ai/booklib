## Finding 1: SkillsBench — First Large-Scale Benchmark Proving Curated Skills Beat Self-Generated Ones

**Maps to:** Core architecture, Spec 2 (eval harness), marketing/positioning
**Impact:** High | **Effort:** Low (insights + benchmark adoption)

**Paper:** [SkillsBench: Benchmarking How Well Agent Skills Work Across Diverse Tasks](https://arxiv.org/abs/2602.12670) (February 2026)
**Source type:** Preprint — multi-author, dedicated benchmark website (skillsbench.ai), 7,308 trajectories logged
**Authors:** Multiple authors (see paper)
**Credibility:** 4/5 — large-scale empirical benchmark (86 tasks, 11 domains, 7 agent-model configurations, 7,308 trajectories), open methodology with deterministic verifiers, leakage audits conducted. Published February 2026. Has companion website and open data.

**Summary:** SkillsBench is the first standardized benchmark for measuring whether agent skills actually help. Each task is evaluated under three conditions: no skills, curated skills, and self-generated skills. Key findings: (1) **Curated skills raise average pass rate by +16.2 percentage points**, but effects vary wildly by domain (+4.5pp for SE to +51.9pp for Healthcare). (2) **Self-generated skills provide zero average benefit** — models cannot reliably author the procedural knowledge they benefit from consuming. (3) **Focused skills with 2–3 modules outperform comprehensive documentation.** (4) **Smaller models with curated skills can match larger models without them.** (5) 16 of 84 tasks show negative skill deltas, meaning badly designed skills actively hurt. Skills use the SKILL.md format with optional resources (scripts, templates, reference docs).

**Why it matters for BookLib:** This is the strongest empirical validation of BookLib's core thesis to date: curated, human-authored skills > auto-generated context. Specific implications:

1. **"2–3 focused modules" finding** directly validates BookLib's XML-tagged structure. Skills with `core_principles` + `anti_patterns` + one focused section beat kitchen-sink documentation. This compounds with the ETH Zurich study (already in IDEAS.md) and the Lulla et al. AGENTS.md study.
2. **Self-generated skills don't work** — this is the anti-thesis to tools that auto-generate context files. BookLib's value proposition (curated expert knowledge from canonical books) is empirically validated.
3. **The benchmark methodology is adoptable.** SkillsBench's three-condition evaluation (no-skill / curated / self-generated) with deterministic verifiers is exactly what Spec 2's eval harness needs. Adapt their task format for BookLib's domain (coding tasks using BookLib skills).
4. **The +16.2pp number is citable** for marketing and README.
5. **Negative deltas on 16/84 tasks** reinforces the `booklib audit` use case — skills need quality gates.

**Actionable implementation:**
- Cite the +16.2pp and "focused 2–3 modules" findings in README and skill authoring guidelines
- Adopt SkillsBench's three-condition evaluation methodology for Spec 2's eval harness in `benchmark/`
- Update `booklib audit` to flag skills with >3 major sections (warn that focused skills outperform comprehensive ones)
- Position in docs: "Curated skills improve task pass rates by 16pp (SkillsBench, 2026). BookLib packages curated expert knowledge — not auto-generated bloat."

## Finding 2: "Malicious Or Not" — Repository Context Slashes Skill Security False Positives from 46.8% to 0.52%

**Maps to:** Spec 1 (`booklib fetch` security), `lib/skill-verifier.js`
**Impact:** High | **Effort:** Medium

**Paper:** [Malicious Or Not: Adding Repository Context to Agent Skill Classification](https://arxiv.org/abs/2603.16572) (March 2026)
**Source type:** Preprint — 5 authors from Austrian research institutions (Florian Holzbauer, David Schmidt, Gabriel Gegenhuber, Sebastian Schrittwieser, Johanna Ullrich)
**Credibility:** 4/5 — largest skill corpus to date (238,180 unique skills from 3 platforms + GitHub), systematic security classification methodology, strong quantitative results. Security research from established institution.

**Summary:** The authors collected 238,180 unique agent skills from three major distribution platforms and GitHub — the largest skill corpus assembled for security analysis. Traditional automated security scanners (pattern-matching for URLs, shell commands, encoded content) flag **46.8% of marketplace skills as malicious** — an absurdly high false positive rate that makes security warnings useless. The paper's key insight: adding **repository context** (repo stars, contributor count, license, CI status, other files in the repo) to the classification pipeline reduces the flagged rate to **0.52%** of skills in malicious-flagged repositories. The approach uses repository-level signals as priors before examining skill content.

**Why it matters for BookLib:** This directly addresses a problem BookLib will face with `booklib fetch`. The SkillTester security probes (already in IDEAS.md) do pattern-matching — which this paper shows produces ~47% false positives. If `booklib fetch` flags half of all third-party skills as dangerous, users will ignore the warnings entirely. The fix: incorporate repository context signals alongside content-based analysis.

**Actionable implementation:**
1. In `lib/skill-verifier.js`, implement a two-tier security classification:
   - **Tier 1 (content scan):** Existing pattern-matching probes (URLs, shell commands, encoded strings, prompt injection) — fast, runs on all skills
   - **Tier 2 (context enrichment):** For skills flagged by Tier 1, check GitHub repo metadata (stars, contributors, license, CI status) before escalating to user warning
2. For `booklib fetch --from=github`, use the GitHub API to pull repo metadata and factor it into the security label
3. Adjust the three-level labeling (safe/caution/unsafe from SkillTester idea in IDEAS.md) to account for repository context:
   - ✅ safe = clean content scan OR flagged content but trusted repo context
   - ⚠️ caution = flagged content + neutral/unknown repo context
   - 🚫 unsafe = flagged content + suspicious repo context (low stars, no CI, single contributor, no license)
4. Cite the 46.8% → 0.52% reduction in security documentation to explain why context-aware scanning matters

**Files:** `lib/skill-verifier.js`, `lib/skill-fetcher.js`, `bin/cli.js`

## Finding 3: Microsoft Harrier-OSS-v1-270M — SOTA Embedding Model with ONNX + Transformers.js Support

**Maps to:** Spec 2 (embedding pipeline), `lib/engine/indexer.js`, `lib/engine/searcher.js`
**Impact:** High | **Effort:** Medium

**Source:** [Microsoft Harrier-OSS-v1-270m](https://huggingface.co/microsoft/harrier-oss-v1-270m) + [ONNX export](https://huggingface.co/onnx-community/harrier-oss-v1-270m-ONNX) (March 30, 2026)
**Source type:** Official model release from Microsoft, open-weight
**Credibility:** 5/5 — Microsoft Research release, SOTA on Multilingual MTEB v2, open weights + ONNX export, documented architecture and training methodology

**Summary:** Microsoft released Harrier-OSS-v1, a family of decoder-only embedding models at three sizes: 270M, 0.6B, and 27B parameters. The 270M variant achieves SOTA on Multilingual MTEB v2 in its size class. Key features: (1) **32,768 token context window** — 64x larger than all-MiniLM-L6-v2's 512 tokens. (2) **ONNX export already available** at `onnx-community/harrier-oss-v1-270m-ONNX`. (3) **Confirmed compatible with Transformers.js** (`@huggingface/transformers`). (4) Uses last-token pooling + L2 normalization. (5) Supports instruction-prefixed queries for task-specific retrieval. (6) Trained via knowledge distillation from larger models.

**Why it matters for BookLib:** BookLib currently uses all-MiniLM-L6-v2 (22M params, 512 token context). Harrier-270M offers three major upgrades:

1. **32K context window enables native late chunking.** The 512-token limit of MiniLM forces aggressive chunking. Harrier-270M can embed an entire SKILL.md file (most are <8K tokens) in a single pass, making late chunking (already in IDEAS.md backlog) trivially implementable — embed whole file, then chunk embeddings.
2. **Instruction-prefixed retrieval** means BookLib can prefix queries with task-specific instructions (e.g., "Retrieve coding best practices for:") for better domain separation.
3. **ONNX + Transformers.js confirmed** means this is a drop-in path for BookLib's existing pipeline.

**Tradeoffs to evaluate:**
- **270M vs 22M params:** ~12x larger model. Needs benchmarking for CLI latency. First-run model download will be larger.
- **Decoder-only architecture:** Different pooling strategy (last-token vs. mean pooling). Requires changes to `indexer.js` embedding logic.
- **Quality vs speed:** The Transformers.js v4 upgrade (already in IDEAS.md) should be completed first to get the 4x speedup baseline, then evaluate Harrier on top of that.

**Actionable implementation:**
1. After completing Transformers.js v4 migration, benchmark Harrier-270M-ONNX vs all-MiniLM-L6-v2 on BookLib's skill retrieval tasks
2. Measure: (a) embedding quality on BookLib's benchmark queries, (b) indexing latency, (c) search latency, (d) model download size
3. If quality wins justify the size increase, implement as an optional `--model=harrier` flag before making it the default
4. With 32K context, implement late chunking: embed full SKILL.md → split token embeddings → mean pool per section
5. Add instruction prefixing to query pipeline in `lib/engine/searcher.js`

**Files:** `lib/engine/indexer.js`, `lib/engine/searcher.js`, `package.json`, `lib/engine/model-config.js` (new)

## Finding 4: AGENTS.md Formalized as Linux Foundation Standard — 60K+ Repos, Official Spec

**Maps to:** Spec 1 (ecosystem alignment), `booklib import/export` (already in IDEAS.md)
**Impact:** Medium | **Effort:** Low (spec alignment check)

**Source:** [Linux Foundation AAIF Announcement](https://www.linuxfoundation.org/press/linux-foundation-announces-the-formation-of-the-agentic-ai-foundation) (December 2025) + [AGENTS.md Specification](https://agents.md/) + [MCP Dev Summit 2026](https://events.linuxfoundation.org/2026/02/24/agentic-ai-foundation-unveils-mcp-dev-summit-north-america-2026-schedule/)
**Source type:** Official Linux Foundation / AAIF announcements
**Credibility:** 5/5 — first-party organizational announcements, verifiable adoption numbers

**Summary:** AGENTS.md has been formalized as an open standard under the Agentic AI Foundation (AAIF), co-founded by Anthropic and OpenAI under the Linux Foundation. Key facts: (1) **60,000+ repos** have adopted AGENTS.md (up from the ~3K repos studied by Galster et al. in the 04/03 report). (2) **Supported by every major AI coding agent**: Amp, Codex, Cursor, Devin, Factory, Gemini CLI, GitHub Copilot, Jules, VS Code. (3) MCP, goose, and AGENTS.md are the three founding projects of AAIF. (4) Projects report **35–55% fewer agent-generated bugs** after implementing AGENTS.md. (5) The spec is deliberately minimal: plain Markdown, no required fields, common sections for setup, code style, testing, architecture, and boundaries.

**Why it matters for BookLib:** The `booklib import/export` feature (P1 in IDEAS.md, from the Galster et al. finding) is now even more urgent. AGENTS.md isn't just a popular convention — it's a Linux Foundation standard backed by every major agent. Specific implications:

1. **The import path is critical.** 60K repos already have AGENTS.md files. `booklib import --from=agents-md` would instantly make BookLib useful in all of them — converting flat context into structured, searchable, progressive-disclosure skills.
2. **The export path positions BookLib as an AGENTS.md authoring tool.** If a developer writes BookLib skills (with XML tags, structured sections), they can `booklib export --format=agents-md` to generate a well-organized AGENTS.md that every agent understands.
3. **MCP + AGENTS.md under one foundation** means BookLib's dual identity (MCP server + skill manager) is perfectly aligned with AAIF's mission.
4. **The MCP Dev Summit (April 2-3, NYC)** is an opportunity for BookLib visibility.

**Actionable implementation:**
- Ensure `booklib import --from=agents-md` parser handles the official spec's common section headings (Setup, Code Style, Testing, Architecture, Boundaries)
- Update the bidirectional conversion feature proposal in IDEAS.md to reference the AAIF standardization and 60K repo number
- Consider submitting BookLib as an AAIF ecosystem project or case study
- Update README competitive positioning: BookLib converts AGENTS.md (Linux Foundation standard, 60K repos) into structured skills with progressive disclosure

## Cross-Cutting Theme

Today's findings paint a picture of the **skill ecosystem maturing from experimentation to standardization and measurement**. SkillsBench provides the first rigorous evidence that curated skills work (+16.2pp) while self-generated ones don't — vindicating BookLib's entire approach. "Malicious Or Not" shows that naive security scanning produces unusable false positive rates, pointing to the need for context-aware verification. Harrier-270M opens a path to dramatically better embeddings with native long-context support. And AGENTS.md's formalization under the Linux Foundation (60K repos) makes the import/export feature even more strategically critical.

The actionable priority chain: (1) cite SkillsBench in positioning materials immediately (zero effort, high impact), (2) complete the AGENTS.md import/export feature (already P1 in IDEAS.md, now more urgent), (3) implement context-aware security scanning for `booklib fetch`, (4) benchmark Harrier-270M after Transformers.js v4 migration.
