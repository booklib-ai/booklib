## What's Worth Building

### Your Indexer Is About to Get 4x Faster — For Free

**The problem it solves:** Every time you run `booklib scan`, the indexer feeds each skill chunk through an embedding model to build the search index. That embedding step is the bottleneck. On a 22-skill library it's tolerable; as BookLib grows toward 258+ discoverable skills, it becomes a real wait.

**How it works:** HuggingFace just released Transformers.js v4 — the same library BookLib already uses for embeddings. The new version ships with a custom ONNX operator called `MultiHeadAttention` that collapses what used to be dozens of separate math operations into one fused step. Think of it like replacing a Rube Goldberg machine with a single lever that does the same job. The result is roughly 4x faster embedding generation for BERT-style models like the MiniLM model BookLib uses. On top of that, the bundle is 53% smaller and now supports WebGPU for users with a GPU, though CPU-only mode keeps working just fine.

**What it means for BookLib:** Faster `booklib scan`, faster `booklib index`, and a snappier CLI experience overall. Users won't need to change anything about how they work — search results stay identical because the model itself hasn't changed, just the runtime executing it. And since no competitor has adopted v4 yet, BookLib would be first.

**What we'd actually build:** Update `package.json` from `@xenova/transformers` to `@huggingface/transformers` v4, change the model ID from `Xenova/all-MiniLM-L6-v2` to `onnx-community/all-MiniLM-L6-v2-ONNX` in the indexer config, and verify that the output embeddings are identical. Half a day to a full day of work.

**Verdict:** ✅ This is the single lowest-risk, highest-impact item on the board. A dependency upgrade from the library's own maintainers with a preserved API. Do it first.

### Science Says: Less Context, Better Results

**The problem it solves:** There's a temptation when building AI skills to dump everything you know into the context window — every rule, every edge case, every caveat. But does more context actually help? Or does it just cost more tokens and confuse the agent?

**How it works:** Researchers at ETH Zurich's SRI Lab — one of the world's top programming languages and security groups — ran a rigorous study across four AI coding agents (including Claude) on hundreds of real software engineering tasks. They tested what happens when you give agents context files (like AGENTS.md or CLAUDE.md) versus letting them work without any. The surprising finding: LLM-generated context files actually *reduced* task success rates while increasing inference costs by over 20%. Agents followed the instructions precisely — using 2.5x more tools, exploring more broadly — but that extra activity didn't translate into better outcomes. The key insight is that skills should contain only **non-inferable details**: things the agent genuinely can't figure out on its own.

**What it means for BookLib:** This is a direct validation of BookLib's structured XML-tagged format. Unlike flat markdown skills (the approach used by skills.sh, Cursor rules, and Copilot instructions), BookLib forces authors to categorize knowledge into discrete tagged sections. That structure makes it possible to inject *only* the high-signal parts — `core_principles` and `anti_patterns` — rather than the entire skill. It's also powerful positioning: BookLib can now credibly claim its format is "research-backed skill design."

**What we'd actually build:** Three things. First, update the skill authoring docs to cite this study and recommend concise, non-inferable content. Second, add a `booklib audit` command that flags skills with oversized sections (token counts per XML tag). Third, add a `--compact` flag to the parser that returns only `core_principles` and `anti_patterns` for contexts where minimal injection is preferred.

**Verdict:** ✅ Actionable today. The guidelines update is a couple hours of writing. The audit command and compact mode are about a day of coding. Together they sharpen every skill in the ecosystem.

### Teach Your Reranker to Quit While It's Ahead

**The problem it solves:** Cross-encoder reranking is the gold standard for search quality — it reads the query and each candidate document together and makes a much more nuanced relevance judgment than vector similarity alone. The problem is it's slow, because it runs a full neural network forward pass for every candidate. SEE (Similarity-based Early Exit) lets the model stop early when it's already confident enough.

**How it works:** Imagine you're reading student essays to rank them. For some essays, you can tell within the first paragraph whether they're excellent or terrible — you don't need to read all five pages. SEE does the same thing with a cross-encoder's internal layers: after each layer, it checks whether the ranking is stable enough to stop. For easy comparisons, it exits early and saves compute. For close calls, it reads all the way through.

**What it means for BookLib:** In theory, this could make a future reranking pipeline significantly faster. In practice, there's a catch — BookLib doesn't have cross-encoder reranking yet (it's planned for Spec 2), and SEE's implementation is PyTorch-only with a GPU requirement. Porting the layer-by-layer early exit logic to BookLib's ONNX/transformers.js stack would require custom graph manipulation that doesn't exist today.

**What we'd actually build:** This is a backlog item, not immediate work. First, BookLib needs basic cross-encoder reranking in Spec 2. Then, someone would need to create ONNX models with per-layer outputs and implement the early-exit logic in JavaScript. That's 2-3 weeks of work total.

**The catch:** The reference implementation requires Docker, NVIDIA GPU runtime, and ~100GB of shared memory. None of that maps to BookLib's "five runtime deps, runs on your laptop" philosophy. This needs significant adaptation work before it's usable.

**Verdict:** ⚠️ The research is top-tier (peer-reviewed at SIGIR 2025), but the implementation gap is real. File it for after Spec 2's hybrid pipeline is stable.

### Don't Dump the Whole Toolbox — Hand Tools Over One at a Time

**The problem it solves:** When an AI agent starts a task, most systems load every available tool into context upfront. That's like handing someone every tool in a hardware store before they've told you what they're building. It wastes context space and can confuse the agent about what's relevant.

**How it works:** The OpenDev project — an open-source Rust-based coding agent — implements what they call **lazy tool discovery**. Instead of loading all tools at startup, the agent discovers them progressively as needed. It also uses **adaptive context compaction**, which progressively compresses older observations as the conversation grows, keeping the most recent and relevant information at full fidelity. Think of it like a photographer's memory: today's shoot is crystal clear, last week's is a strong impression, last month's is just the highlights.

**What it means for BookLib:** BookLib's XML-tagged skill format already has natural disclosure layers — metadata, core principles, examples, anti-patterns. A `--progressive` flag could output skills in stages: metadata first, then core principles on demand, then the full skill. This aligns with the ETH Zurich finding that less context performs better. The combination of these two studies tells a coherent story: deliver knowledge in layers, not dumps.

**What we'd actually build:** A `booklib context --progressive` flag that outputs three disclosure levels using the existing parser. This is about a day of work. The more ambitious idea — modifying BookLib's MCP server to deliver skills in stages using non-standard response fields — carries adoption risk and should wait.

**The catch:** The OpenDev paper is a system description, not a controlled experiment. It shows what they built, not proof that lazy discovery outperforms eager loading in isolation. The design pattern is sound, but the evidence is architectural, not experimental.

**Verdict:** ⚠️ Good design wisdom, but implement it through BookLib's existing mechanisms (the `--compact` flag from Finding 2, progressive parser modes) rather than non-standard MCP extensions.
