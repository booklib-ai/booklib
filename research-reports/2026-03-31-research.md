## Finding 1: Late Chunking — Contextual Chunk Embeddings Without LLM Cost

**Maps to:** Spec 2 — `lib/engine/indexer.js`
**Impact:** High | **Effort:** Medium

**Paper:** [Late Chunking: Contextual Chunk Embeddings Using Long-Context Embedding Models](https://arxiv.org/abs/2409.04701) (updated July 2025)

**Key insight:** Late chunking reverses the standard chunk-then-embed pipeline. Instead, it embeds the *entire document* through a long-context embedding model first, then chunks the token-level embeddings and applies mean pooling per chunk. Each chunk embedding therefore captures full document context — achieving +24.47% retrieval improvement on the LongEmbed benchmark vs. standard 512-token chunking — *without* requiring an LLM call per chunk (unlike Anthropic's contextual retrieval).

**Why it matters for BookLib:** Your `thoughts.md` already flags Anthropic contextual retrieval as high-priority, but it requires an LLM call per chunk at index time — expensive and slow for a CLI tool. Late chunking achieves similar contextual awareness at pure embedding cost. The catch: it requires a long-context embedding model. `Xenova/jina-embeddings-v2-small-en` supports 8192 tokens and runs in transformers.js. This would be a drop-in enhancement to `BookLibIndexer.indexDirectory()` — group chunks by skill file, embed the full file, then split.

**Competitive edge:** No CLI-based skill tool currently uses late chunking. This would be a genuine first-mover advantage over skills.sh and agentskills.io's flat indexing.

## Finding 2: SRAG — Structured Metadata Enrichment for Chunks

**Maps to:** Spec 2 — `lib/engine/indexer.js`, `lib/engine/parser.js`
**Impact:** High | **Effort:** Medium

**Paper:** [SRAG: RAG with Structured Data Improves Vector Retrieval](https://arxiv.org/html/2603.26670) (January 2026)

**Key insight:** SRAG prepends structured metadata — topics, semantic tags, knowledge graph triples, and chunk type classifiers — directly into the text before embedding. This gives the embedding model explicit structural signals alongside the semantic content. The result: significantly better precision on domain-specific retrieval where chunks from different domains share vocabulary.

**Why it matters for BookLib:** This directly addresses the noise problem your `thoughts.md` identifies — "a TypeScript-specific query competes with Kotlin and DDD chunks." BookLib's `parseSkillFile()` already extracts XML tag types (`core_principles`, `anti_patterns`, `framework`). If indexer.js prepends structured prefixes like `[skill:effective-kotlin] [type:anti_patterns] [tags:null-safety,kotlin]` before embedding each chunk, the vector space naturally separates by domain. This is complementary to the two-stage retrieval idea already in your roadmap, and much cheaper to implement.

**Competitive edge:** BookLib's XML-tagged skill format is *uniquely positioned* for this — you already have the structured metadata that SRAG requires. Flat markdown skills on skills.sh don't.

## Finding 3: Codified Context Infrastructure — The Academic Validation of BookLib's Architecture

**Maps to:** Spec 3 + marketing/positioning
**Impact:** High | **Effort:** Low

**Paper:** [Codified Context: Infrastructure for AI Agents in a Complex Codebase](https://arxiv.org/abs/2602.20478) (February 2026)

**Key insight:** This paper formalizes the idea that documentation is load-bearing infrastructure for AI agents. It describes a three-component system: (1) hot-memory constitution (conventions + orchestration protocols), (2) specialized domain-expert agents, and (3) cold-memory knowledge base of on-demand spec documents. Built on a 108K-line C# codebase. The paper explicitly argues that AI agents lose coherence across sessions and repeat mistakes without persistent codified context.

**Why it matters for BookLib:** This is essentially an academic validation of what BookLib already does — skills as hot memory, reviewer agents as domain experts, and the knowledge graph as cold memory. The paper's taxonomy maps almost 1:1 to BookLib's architecture. This is less about implementation and more about positioning: cite this paper in your README, RESEARCH.md, and any talks. Use their terminology ("codified context infrastructure") when describing BookLib to academic and enterprise audiences. It legitimizes the entire approach.

**Competitive edge:** BookLib can position itself as the open-source implementation of the codified context pattern described in peer-reviewed research. No other tool in the ecosystem is doing this.

## Finding 4: KG²RAG — Knowledge Graph-Guided Chunk Expansion

**Maps to:** Spec 3 — `lib/engine/graph.js`, `lib/engine/searcher.js`
**Impact:** Medium | **Effort:** Medium

**Paper:** [Knowledge Graph-Guided Retrieval Augmented Generation (KG²RAG)](https://arxiv.org/abs/2502.06864) (February 2025)

**Key insight:** KG²RAG uses a knowledge graph to provide fact-level relationships *between* chunks. After initial semantic retrieval, it performs KG-guided chunk expansion (following edges to pull in related chunks) and KG-based chunk organization (ordering chunks by graph distance to maintain coherence). This is distinct from GraphRAG's global summarization approach — it works at the chunk level and preserves granularity.

**Why it matters for BookLib:** Your Spec 3 already plans `--graph` augmented retrieval that follows edges one hop. KG²RAG's contribution is the *organization* step — ordering expanded results by graph distance so the LLM receives them in a coherent sequence rather than a flat ranked list. This is a small but meaningful enhancement to the `booklib context` command's output. The implementation would live in `graph-injector.js` — after graph expansion, sort by hop distance, then by edge type priority (`applies-to` > `see-also` > `extends`).

**Competitive edge:** No CLI skill tool has graph-organized retrieval output. Combined with BookLib's existing graph infrastructure, this makes the knowledge graph a retrieval *quality* tool, not just a navigation tool.

## Finding 5: MCP Server Cards + .well-known Discovery

**Maps to:** Spec 1 — `docs/.well-known/`, CI pipeline
**Impact:** Medium | **Effort:** Low

**Source:** [MCP 2026 Roadmap — Server Cards](https://blog.modelcontextprotocol.io/posts/2026-mcp-roadmap/)

**Key insight:** The MCP project's 2026 roadmap introduces "MCP Server Cards" — a standard for exposing structured server metadata via a `.well-known` URL so that registries, crawlers, and agents can discover server capabilities without connecting. This is being built into the official MCP spec alongside streamable HTTP transport improvements for remote MCP servers.

**Why it matters for BookLib:** Spec 1 already plans a `/.well-known/skills/` endpoint for skill discovery. The MCP Server Card format is converging on the same `.well-known` pattern. If `booklib build-wellknown` also generates an MCP Server Card alongside the skill index, BookLib's MCP server becomes discoverable by the entire MCP ecosystem — not just skill-specific tools. This is a low-effort addition to the existing CI pipeline that dramatically increases surface area.

**Competitive edge:** Being early to MCP Server Cards means BookLib shows up in MCP registries and agent discovery tools automatically. First movers here get indexed first.

## Finding 6: Security Alert — Malicious Skills in Community Hubs

**Maps to:** Spec 1 — `lib/skill-fetcher.js`, `lib/registry/`
**Impact:** High | **Effort:** Medium

**Source:** [Agent Skills Guide 2026](https://serenitiesai.com/articles/agent-skills-guide-2026) + ecosystem reports

**Key insight:** In February 2026, security researchers identified 341 malicious skills on community hubs. As the ecosystem has grown to 100K+ installs across 16+ agents, skill supply chain security has become a real concern. Skills can contain prompt injection attacks, exfiltration instructions, or malicious hook commands.

**Why it matters for BookLib:** BookLib's `booklib fetch` and discovery engine pull skills from external sources (GitHub repos, registries, npm). Currently there's no verification layer. Adding a skill verification/sandboxing step — even a simple static analysis pass that checks for suspicious patterns (URLs in skill text, shell commands, base64-encoded content) — would differentiate BookLib as the *security-conscious* choice. This maps to `lib/skill-fetcher.js` and potentially a new `lib/skill-verifier.js`.

**Competitive edge:** Neither skills.sh nor agentskills.io currently offer client-side skill verification. Being the first tool to flag potentially malicious skills at fetch time is a strong trust signal, especially for enterprise adoption.
