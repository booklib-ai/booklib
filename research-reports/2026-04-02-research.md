## 1. Efficient Re-ranking with Cross-encoders via Early Exit (SEE)

**Paper:** [Efficient Re-ranking with Cross-encoders via Early Exit](https://dl.acm.org/doi/10.1145/3726302.3729962)  
**Authors:** Busolin, Lucchese, Nardini, Orlando, Perego, Trani, Veneri  
**Venue:** SIGIR 2025 (peer-reviewed) | **Source type:** Peer-reviewed, top-tier IR venue  
**Code:** https://github.com/veneres/SEE-SIGIR25

**Summary:** Proposes an early-exit strategy for cross-encoder reranking that lets the model terminate transformer layer traversal early for "easy" query-document pairs while fully processing harder ones. This dramatically reduces latency without meaningful quality loss — directly applicable to BookLib's planned Spec 2 cross-encoder reranking step where CPU budget is tight.

**Maps to:** Spec 2 — Retrieval Quality (cross-encoder reranking)  
**Actionable implementation:** Port the early-exit logic to the planned reranker in `src/retrieval/reranker.js`. Use ONNX-exported cross-encoder with layer-wise confidence thresholds — skip remaining layers when score delta drops below ε. Could halve reranking latency on CPU for the top-K candidates.  
**Impact:** High | **Effort:** Medium

## 2. Towards Practical GraphRAG: Efficient KG Construction and Hybrid Retrieval at Scale

**Paper:** [Towards Practical GraphRAG](https://arxiv.org/abs/2507.03226)  
**Authors:** Min, Bansal, Pan, Keshavarzi, Mathew, Kannan  
**Venue:** arXiv preprint (6 authors, enterprise research team) | **Source type:** Preprint, multi-author industry team  
**Date:** July 2025 (revised Dec 2025)

**Summary:** Demonstrates that dependency-parsing-based KG construction achieves 94% of LLM-based extraction quality (61.87% vs 65.83%) at a fraction of the cost. Their hybrid retrieval fuses vector similarity with graph traversal via RRF, maintaining separate embeddings for entities, chunks, and relations. Shows 15% improvement over vanilla vector baselines on enterprise code migration datasets.

**Maps to:** Spec 3 — Knowledge Graph + Research  
**Actionable implementation:** Adopt dependency-parsing (not LLM) for KG construction in `src/graph/`. Use their multi-granular embedding approach: separate vector indices for graph nodes (concepts), edges (relationships), and chunks (skill content). Fuse with RRF in the hybrid retrieval pipeline alongside BM25 + vector from Spec 2.  
**Impact:** High | **Effort:** High

## 3. Context is Gold: ConTEB Benchmark and InSeNT Training for Contextual Embeddings

**Paper:** [Context is Gold to find the Gold Passage](https://arxiv.org/abs/2505.24782)  
**Authors:** Conti, Faysse, Viaud, Bosselut, Hudelot, Colombo  
**Venue:** arXiv preprint (6 authors, EPFL + Illuin Technology) | **Source type:** Preprint, academic + industry  
**Date:** May 2025

**Summary:** Introduces ConTEB, a benchmark showing SOTA embedding models fail when document-wide context is needed. Their InSeNT (In-Sequence Negative Training) method, combined with late chunking pooling, significantly improves contextual chunk embeddings without degrading base model quality. Chunks become robust to suboptimal chunking strategies.

**Maps to:** Spec 2 — Retrieval Quality (embedding pipeline)  
**Actionable implementation:** BookLib's skills are chunked markdown with XML tags — a perfect case for contextual embeddings. Investigate applying late chunking via long-context models (jina-embeddings-v3 or nomic-embed) in `src/retrieval/embedder.js` so each skill section retains awareness of the full skill document. If switching from all-MiniLM-L6-v2 is too heavy, the InSeNT post-training approach could be applied to fine-tune the existing model.  
**Impact:** High | **Effort:** Medium

## 4. Transformers.js v4: WebGPU Runtime, New Package, 4x Embedding Speedup

**Paper:** [Transformers.js v4 Preview: Now Available on NPM](https://huggingface.co/blog/transformersjs-v4)  
**Authors:** Hugging Face team (Joshua Lochner et al.)  
**Source type:** Official project documentation / release announcement  
**Date:** February 9, 2026

**Summary:** Major release migrating from `@xenova/transformers` to `@huggingface/transformers`. Key improvements: WebGPU runtime rewritten in C++ delivers ~4x speedup for BERT-based embedding models via specialized ONNX operators; new ModelRegistry API for cache management and dtype selection; 10x faster build times; 53% smaller web bundle. Supports models >8B params. New standalone `@huggingface/tokenizers` (8.8kB gzipped).

**Maps to:** Spec 2 — Retrieval Quality (local ML inference) + Core architecture  
**Actionable implementation:** BookLib currently depends on `@xenova/transformers`. Migration path: (1) swap to `@huggingface/transformers` in `package.json`, (2) update import paths across `src/retrieval/`, (3) leverage ModelRegistry API for smarter model caching in `src/retrieval/model-manager.js`, (4) test WebGPU acceleration on supported hardware for embedding generation. The 4x speedup on BERT models directly benefits all-MiniLM-L6-v2 inference.  
**Impact:** High | **Effort:** Low-Medium (mostly import path changes + testing)

## 5. On the Impact of AGENTS.md Files on AI Coding Agent Efficiency

**Authors:** Lulla, Mohsenimofidi, Galster, Zhang, Baltes, Treude  
**Venue:** arXiv preprint (6 authors, multi-university) | **Source type:** Preprint, academic multi-institution  
**Date:** January 2026 (revised March 2026)

**Summary:** First empirical study measuring AGENTS.md file impact across 10 repos and 124 PRs. With AGENTS.md present: 28.64% median runtime reduction, 16.58% fewer output tokens, with comparable task completion. Validates that structured agent configuration files provide measurable efficiency gains — directly relevant to BookLib's skill format as agent context.

**Maps to:** Spec 1 — Ecosystem Alignment + Core skill format

Consider aligning skill metadata fields with the conventions this study found most effective. Could inform `booklib init` output format and the `.well-known` skill manifest spec.  
**Impact:** Medium | **Effort:** Low

## 6. Codified Context: Three-Tier Infrastructure for AI Agents in Complex Codebases

**Paper:** [Codified Context: Infrastructure for AI Agents in a Complex Codebase](https://arxiv.org/abs/2602.20478)  
**Authors:** Aristidis Vasilopoulos  
**Venue:** arXiv preprint | **Source type:** Preprint, single author but with open-source companion repo + 283-session empirical study  
**Date:** February 24, 2026  
**Code:** https://doi.org/10.5281/zenodo.18746623

**Summary:** Presents a three-tier knowledge architecture for coding agents: (1) hot-memory constitution (always loaded — conventions, retrieval hooks, orchestration), (2) 19 specialized domain-expert agents with embedded knowledge, (3) cold-memory knowledge base of 34 spec documents served via MCP retrieval. Tested across 283 sessions on a 108K-line codebase. Demonstrates how structured context propagation prevents cross-session failures.

**Maps to:** Spec 3 — Knowledge Graph + Research + Core architecture  
**Actionable implementation:** BookLib's architecture already mirrors this pattern partially (skills = cold memory, hooks = hot memory). Formalize the three-tier model: (1) make `booklib context` output serve as hot-memory constitution, (2) use the knowledge graph (Spec 3) as the cold-memory MCP retrieval layer, (3) design skill profiles as domain-expert agent configurations. The MCP retrieval service pattern is directly portable to BookLib's MCP server in `src/mcp/`.  
**Impact:** Medium | **Effort:** Medium

## 7. Building Effective AI Coding Agents for the Terminal: OPENDEV

**Paper:** [Building Effective AI Coding Agents for the Terminal](https://arxiv.org/abs/2603.05344)  
**Authors:** Nghi D. Q. Bui  
**Venue:** arXiv preprint | **Source type:** Preprint, single author, open-source Rust implementation  
**Date:** March 5, 2026 (revised March 13, 2026)

**Summary:** Introduces OPENDEV, a Rust-based terminal coding agent. Key architectural patterns relevant to BookLib: (1) **lazy tool discovery** — tools are discovered on-demand rather than pre-loaded, reducing prompt bloat; (2) **workload-specialized model routing** — different task types are directed to appropriately-sized models; (3) **adaptive context compaction** — older observations are progressively compressed to prevent context degradation; (4) **event-driven system reminders** to counteract instruction fade-out.

**Maps to:** Spec 1 — Ecosystem Alignment (agent support) + Core architecture  
**Actionable implementation:** The lazy tool discovery pattern maps directly to BookLib's PreToolUse hook — skills should be surfaced on-demand based on detected intent, not bulk-loaded. The model routing concept could inform a future `booklib route` command that selects optimal skill subsets based on task type. Adaptive context compaction is relevant for long `booklib context` outputs — consider a summarization/priority mode for large skill sets.  
**Impact:** Medium | **Effort:** Low (pattern adoption, not new code)
