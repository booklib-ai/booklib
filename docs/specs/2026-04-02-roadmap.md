# BookLib Roadmap — Specific Specs
*April 2, 2026*

## Positioning

BookLib's unique space: **expert knowledge + personal project context + autonomous consolidation + multi-tool MCP.** Not competing with Mem0 on conversation memory. Not competing with Obsidian on note-taking. Competing on: making AI agents genuinely smarter with knowledge they can't get from training data.

---

## Phase 1: Polish What's Built (1-2 days)

Things that are 90% done and need finishing.

### Spec P1-1: Transformers.js v4 Upgrade
**What:** Update model ID from `Xenova/all-MiniLM-L6-v2` to `onnx-community/all-MiniLM-L6-v2-ONNX`. Import paths already updated.
**Why:** ~4x faster indexing. Free performance win.
**Files:** `lib/engine/indexer.js`, `lib/engine/searcher.js`, `lib/engine/reranker.js`
**Acceptance:** Indexing benchmark shows measurable speedup. All tests pass. Same search results.
**Effort:** 2 hours

### Spec P1-2: Remove Legacy Content Dump Code
**What:** Delete `_extractBlocks()` and old `_render()` from `lib/project-initializer.js`. They're behind `legacy: true` flag, no longer called by any path.
**Why:** Dead code. Prevents accidental 10,000-line config file generation.
**Files:** `lib/project-initializer.js`
**Acceptance:** `grep -r '_extractBlocks\|legacy.*true' lib/` returns nothing. All tests pass.
**Effort:** 30 minutes

### Spec P1-3: Compact Parser Mode
**What:** Add `--compact` option to `parseSkillFile()` that returns only `core_principles` + `anti_patterns` XML tags, skipping `examples`, `summary`, and other sections.
**Why:** ETH Zurich research says less context = better performance. MCP responses should be minimal.
**Files:** `lib/engine/parser.js`
**Acceptance:** `parseSkillFile(content, path, { compact: true })` returns fewer chunks. Regular mode unchanged.
**Effort:** 2 hours

### Spec P1-4: Fix Website Homepage
**What:** Update `docs/index.html` to match current positioning — remove outdated feature claims, update tool names, add graph search and dream service teaser.
**Why:** Website is the first thing people see. Still claims PreToolUse hook in some places.
**Files:** `docs/index.html`
**Acceptance:** No false claims. Current features accurately described.
**Effort:** 2 hours

## Phase 2: Personal Graph Layer (3-5 days)

The foundation for the dream service. BookLib currently has only project-scoped knowledge. This adds a per-user layer.

### Spec P2-1: Personal Observation Store
**What:** New directory `~/.booklib/personal/observations/`. Every `lookup` MCP call logs `{ query, results_count, timestamp, file }` as an observation. Every `remember` call logs `{ title, type, timestamp }`. Every correction logs `{ text, timestamp }`.
**Why:** Feeds the dream service. Can't consolidate patterns without raw observations.
**Files:** New `lib/engine/personal-store.js`. Modify `bin/booklib-mcp.js` to log after each tool call.
**Acceptance:** After a session with 5 lookups and 2 remembers, `~/.booklib/personal/observations/` contains 7 entries. Entries are append-only JSONL.
**Effort:** 1 day

### Spec P2-2: Pattern Detection (Fast Mode)
**What:** Analyze observations to detect repeating patterns. Rule-based: "query X appeared 3+ times in 7 days → pattern." "File Y was looked up with topic Z every time → association."
**Why:** Patterns become preferences. Preferences become pre-fetched context.
**Files:** New `lib/engine/pattern-detector.js`
**Acceptance:** Given 10 observations with 3 repeated queries, detects 1 pattern. Given 5 observations all mentioning the same file, detects 1 file-topic association.
**Effort:** 1 day

### Spec P2-3: Personal Preferences
**What:** Patterns promoted to preferences after threshold (e.g., pattern observed 5+ times). Stored in `~/.booklib/personal/preferences.json`. Preferences modify `lookup` behavior — pre-boost results matching preferences.
**Why:** The agent learns your habits. "You always check error handling before reviews" → error handling results rank higher automatically.
**Files:** `lib/engine/personal-store.js` (extend), `lib/engine/searcher.js` (preference boost)
**Acceptance:** After preference "error handling before code review" is created, `lookup("code review")` returns error handling results higher than without preference.
**Effort:** 1 day

### Spec P2-4: Morning Briefing
**What:** New MCP tool field in `brief` response: `personal` section with patterns, pre-fetched context, and pending suggestions.
**Why:** When the agent starts a session, it knows what YOU typically need based on your history.
**Files:** `bin/booklib-mcp.js` (extend `brief` handler)
**Acceptance:** `brief` response includes `personal.patterns` array and `personal.pre_fetched` array when personal data exists. Empty gracefully when no data.
**Effort:** 0.5 day

## Phase 3: Dream Service — Manual (2-3 days)

The `booklib dream` command. Manual trigger first. Daemon later.

### Spec P3-1: Project Graph Analysis (Fast Mode)
**What:** `booklib dream` command that analyzes the project knowledge graph and produces a report: duplicates (exact title match), orphans (zero edges), staleness (no activity in 30 days), potential contradictions (two decision nodes about the same component).
**Why:** Knowledge degrades over time. The dream service detects this.
**Files:** New `lib/engine/dream.js`. Modify `bin/booklib.js` (add `case 'dream':`).
**Acceptance:** Given a graph with 2 duplicate titles, 3 orphan nodes, and 1 stale node, report lists all of them.
**Effort:** 1 day

### Spec P3-2: Personal Graph Consolidation (Fast Mode)
**What:** `booklib dream` also consolidates personal observations. Merges duplicate observations, promotes frequent patterns to preferences, prunes observations older than 30 days.
**Why:** Personal data grows unbounded. Consolidation keeps it useful.
**Files:** `lib/engine/dream.js` (extend)
**Acceptance:** Given 50 observations with 10 duplicates and 5 older than 30 days, consolidation produces 35 observations + 2 new patterns.
**Effort:** 1 day

### Spec P3-3: Dream Report with Apply
**What:** `booklib dream` outputs a report. `booklib dream --apply` lets the user select which suggestions to execute. For project graph: merge duplicates, add missing edges, flag contradictions. For personal graph: auto-applied (no approval needed).
**Why:** Project graph changes need approval. Personal changes don't.
**Files:** `lib/engine/dream.js`, `bin/booklib.js`
**Acceptance:** Running `booklib dream --apply` with 2 duplicate suggestions, selecting one, results in nodes merged and edges preserved.
**Effort:** 1 day

### Spec P3-4: Dream MCP Tool
**What:** New MCP tool `dream` — the agent can trigger knowledge consolidation. Returns the report as structured JSON so the agent can present it to the user.
**Why:** The agent should be able to suggest "your knowledge graph needs attention" and handle it.
**Files:** `bin/booklib-mcp.js`
**Acceptance:** Calling `dream` MCP tool returns structured report with `project_suggestions` and `personal_consolidated` arrays.
**Effort:** 0.5 day

## Phase 4: Dream Daemon (2-3 days)

Background service that runs autonomously.

### Spec P4-1: Platform-Specific Daemon Installer
**What:** `booklib dream --install-daemon` installs a background service. macOS: LaunchAgent. Linux: systemd user service. Runs `booklib dream --auto` every 6 hours.
**Why:** Autonomous consolidation without user remembering to run it.
**Files:** New `lib/daemon/installer.js`. Modify `bin/booklib.js`.
**Acceptance:** After install, `launchctl list | grep booklib` (macOS) or `systemctl --user status booklib-dream` (Linux) shows the service. It runs and produces dream logs.
**Effort:** 1 day

### Spec P4-2: Dream Auto Mode
**What:** `booklib dream --auto` mode that runs silently — consolidates personal graph (autonomous), generates project report (saved to `~/.booklib/dream/project-report.md`), logs actions to `~/.booklib/dream/dream-log.jsonl`.
**Why:** The daemon needs a non-interactive mode. No prompts, no output, just files.
**Files:** `lib/engine/dream.js` (extend)
**Acceptance:** Running `booklib dream --auto` produces no stdout, writes report to file, logs actions.
**Effort:** 0.5 day

### Spec P4-3: Wizard Daemon Setup
**What:** `booklib init` asks whether to install the dream daemon. Three options: every 6 hours (recommended), nightly only, manual only.
**Why:** Get the daemon installed during first setup, not as a separate step.
**Files:** `lib/wizard/index.js`
**Acceptance:** Wizard shows the daemon option. Selecting "every 6 hours" installs the LaunchAgent/systemd service.
**Effort:** 0.5 day

### Spec P4-4: Dream Status in Doctor
**What:** `booklib doctor` shows dream daemon status — running/stopped, last run time, pending project suggestions, personal patterns count.
**Why:** Doctor is the health dashboard. Dream service is part of health.
**Files:** `lib/engine/doctor.js`
**Acceptance:** `booklib doctor` includes a "Dream service" section with status and last run time.
**Effort:** 0.5 day

## Phase 5: Conversation Intelligence (2-3 days)

Make the agent smarter about capturing knowledge from conversations.

### Spec P5-1: Smart Capture Detection
**What:** Update `remember` MCP tool description to include detection patterns: "Also capture when the user says 'we decided', 'from now on', 'the rule is', 'always do X', 'never do Y'." The agent recognizes decision-language and proactively calls `remember`.
**Why:** Most knowledge is lost because nobody explicitly says "remember this." The agent should recognize implicit decisions.
**Files:** `bin/booklib-mcp.js` (update tool description)
**Acceptance:** Tool description includes decision-language triggers.
**Effort:** 1 hour

### Spec P5-2: Session Summary Capture
**What:** New MCP tool `summarize_session` — at the end of a session, the agent calls this with a summary of key decisions and insights from the conversation. BookLib creates nodes + auto-links for each.
**Why:** Captures session knowledge that would otherwise be lost when the context window closes.
**Files:** `bin/booklib-mcp.js` (new tool handler)
**Acceptance:** Calling `summarize_session` with 3 insights creates 3 nodes, each auto-linked.
**Effort:** 1 day

### Spec P5-3: Decision vs Insight Classification
**What:** When `remember` is called, BookLib analyzes the title/content to auto-classify as `decision` (higher priority, shown first in results) vs `insight` (supporting knowledge) vs `note` (general). Decisions get `applies-to` edges to components. Insights get `see-also` edges.
**Why:** Not all captured knowledge is equal. Decisions are constraints. Insights are suggestions. The search should prioritize differently.
**Files:** `lib/engine/auto-linker.js` (extend with classification logic)
**Acceptance:** "We will use PostgreSQL for all services" classified as `decision`. "PostgreSQL handles JSON well" classified as `insight`.
**Effort:** 1 day

## Phase 6: Graph Visualization (3-5 days)

Visual representation of the knowledge graph.

### Spec P6-1: Graph Data Export
**What:** `booklib graph --json` exports the full graph (nodes + edges) as a single JSON file suitable for visualization libraries (D3, vis.js, sigma.js).
**Why:** Foundation for visualization. Also useful for external tools.
**Files:** New `lib/engine/graph-export.js`. Modify `bin/booklib.js`.
**Acceptance:** JSON output contains `{ nodes: [...], edges: [...] }` with id, title, type, position-neutral format.
**Effort:** 0.5 day

### Spec P6-2: Local Web Visualizer
**What:** `booklib graph --open` starts a local HTTP server and opens a browser with a force-directed graph visualization. Nodes are colored by type (decision=red, insight=blue, component=green). Edges show relationship type on hover. Click a node to see its content.
**Why:** The screenshot that gets 8K likes on Instagram. Also genuinely useful for exploring knowledge.
**Files:** New `lib/visualizer/` directory with HTML + JS (bundled, no build step). Modify `bin/booklib.js`.
**Acceptance:** Running `booklib graph --open` opens browser with interactive graph. Works offline.
**Effort:** 2-3 days

### Spec P6-3: Graph Diff View
**What:** `booklib graph --diff` shows what changed since last dream run — new nodes, new edges, suggested merges. Visual diff of the knowledge graph.
**Why:** Understand how your knowledge evolves. See what the dream service suggests.
**Files:** `lib/visualizer/` (extend)
**Acceptance:** New nodes shown in green, dream suggestions shown in yellow, contradictions in red.
**Effort:** 1 day

## Phase 7: Ecosystem (ongoing)

### Spec P7-1: MCP Server Cards
**What:** Generate `.well-known/mcp/server-card.json` alongside existing `.well-known/skills/`. Makes BookLib's MCP server discoverable by the MCP ecosystem.
**Files:** `lib/well-known-builder.js`, CI pipeline.
**Effort:** 0.5 day

### Spec P7-2: Obsidian Export
**What:** `booklib export --obsidian <path>` writes all knowledge nodes as Obsidian-compatible markdown with `[[wikilinks]]` for edges. One-way sync.
**Why:** Users with Obsidian get the graph visualization for free. Lowers adoption barrier.
**Files:** New `lib/export/obsidian.js`
**Effort:** 1 day

### Spec P7-3: Import from Obsidian/Markdown
**What:** `booklib import <path>` reads a directory of markdown files, creates knowledge nodes, auto-links them. Supports `[[wikilinks]]` as edge hints.
**Why:** Users bring existing knowledge into BookLib without re-typing.
**Files:** New `lib/import/markdown.js`
**Effort:** 1 day
