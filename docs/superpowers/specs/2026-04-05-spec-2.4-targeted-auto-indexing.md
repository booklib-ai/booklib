# Spec 2.4: Targeted Auto-Indexing

> **Status:** Deferred — depends on reliable per-API docs resolution

## Problem

When BookLib detects an unknown import (Spec 2.2), it suggests `booklib connect <url>`. 
But the user must manually find and index the right docs. Auto-indexing the homepage 
or full docs site produces noise — Spring Boot docs are thousands of pages, most 
irrelevant to the specific API being used.

## Requirements

Index **only** the docs pages relevant to the specific APIs detected as unknown.
Must work reliably across ecosystems without producing junk.

## Why This Is Hard

- Not all docs sites are crawlable (SPAs, auth-gated, bot-blocked)
- API naming differs from import naming (`@RestController` vs "Spring MVC Controllers")
- No standard docs structure across frameworks
- Crawling + embedding is slow (13h for 26K chunks on CPU)
- README approach fails: thin for frameworks, model already knows pre-training READMEs

## Possible Approaches (to evaluate)

### A: Per-API docs page resolution
Resolve the specific docs page for each flagged API. Requires knowing docs URL 
patterns per package — not scalable without a registry of patterns.

### B: Docs site search API
Many docs sites have search (Algolia, DocSearch). Query the docs search with the 
API name, get the specific page URL, index just that page.

### C: Package-provided doc manifests
Emerging pattern: packages ship a `docs.json` or MCP server with structured API docs. 
Index from structured source instead of crawling HTML.

### D: TypeScript definitions as API reference
For npm packages: `.d.ts` files have function signatures, parameters, JSDoc comments. 
Machine-readable, always accurate, available from npm. Limited to JS/TS ecosystem.

### E: Post-training vs pre-training strategy
- Post-training packages: broader indexing makes sense (model knows nothing)
- Pre-training packages: only index changelog/migration (model knows the basics)
- Combine with gap detection (Spec 2.1) to determine indexing depth

## Dependencies

- Spec 2.1 (Gap Detection) — post-training classification
- Spec 2.2 (Import Detection) — knows which specific APIs are unknown
- WebConnector — crawling infrastructure exists
- Quality gate (`assessQuality`) — content validation exists

## Validation Criteria

Before building, validate with 10 real-world packages across 3+ ecosystems:
1. Can we reliably find the right docs page for a specific API?
2. Does the indexed content actually help the model generate better code?
3. Is it faster than manual `booklib connect`?

If any answer is "no" for >30% of cases, defer further.
